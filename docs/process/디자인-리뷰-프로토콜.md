# AI 기반 UI — 생성 및 디자인 리뷰 프로토콜

*`design.md`(디자인 독트린) 및 `@enhans-jooyeon/synapse`(하네스)의 짝 문서입니다. 이 문서는 **프로세스** 독트린입니다. 저장소에 함께 두고, 다른 산출물처럼 버전 관리하고 diff합니다.*

---

## 1. 목적과 범위

비디자이너(엔지니어, PM)가 Synapse 하네스 아래에서 생성형 AI로 프로덕션 시드용 UI를 만들고, 디자이너가 그 결과를 리뷰합니다. 이 프로토콜은 **누가, 어떤 순서로, 무엇을 책임지는가**를 정의합니다. 특히 **사람이 책임져야 하는 것과 책임지지 말아야 하는 것**을 분명히 나눕니다.

사용자에게 보이는 화면을 렌더링하는 모든 UI 변경에 적용합니다. 시각 산출물이 없는 로직/카피만의 변경에는 적용하지 않습니다.

---

## 2. 모든 것의 전제가 되는 하나의 원칙

> **하네스가 규정 준수를 책임지고, 사람은 판단만 리뷰한다.**

기계가 검사할 수 있는 것 — 토큰 사용, 컴포넌트 출처, 대비, 필수 상태 커버리지, prop 유효성 — 은 전부 **게이트**이지 리뷰 항목이 아닙니다. 디자이너가 보기 **전에** 반드시 green이어야 합니다. 디자이너가 spacing 값이나 하드코딩된 hex, 혹은 `Button`을 다시 만든 건 아닌지를 손으로 확인하고 있다면, 그건 하네스에 구멍이 있는 것이고, 해법은 리뷰 단계를 추가하는 게 아니라 그 구멍을 막는 것입니다.

도구 자유(원하는 생성 도구를 쓰는 것)가 성립하는 것도 오직 이 원칙 덕분입니다. `design.md`는 첫 결과물의 품질을 높여줄 뿐 아무것도 강제하지 않습니다. **강제하는 층은 CI입니다.** 이 점을 팀에 명확히 말해 두세요. "가이드를 읽었다"가 "규정을 준수했다"와 같지 않도록.

## 3. 역할

| 역할 | 담당 | 책임 |
|---|---|---|
| **작성자(Author)** | UI를 생성하는 PM–엔지니어 페어 | PRD 작성, 생성, 모든 게이트 통과, 리뷰 가능한 패키지 오픈 |
| **리뷰어(Reviewer)** | 디자이너 | 판단 전용 리뷰, 3가지 결과 중 하나, 범위 내 표면 수정 |
| **하네스 오너** | Synapse/디자인 시스템 오너 | 게이트·토큰·variant·참조 패턴 최신 유지, 피드백 루프 트리아지 |
| **머지 오너** | 지연 (FE 테크리드) | 통합과 일정, 승인된 작업만 머지 |

작성자는 페어일 수 있으나, 제출 1건당 **책임지는 사람 한 명**을 지정합니다.

## 4. PRD — 생성과 리뷰의 단일 출발점

이 프로세스에서 PRD는 두 가지 역할을 동시에 합니다.

1. **생성의 씨앗**: 작성자가 LLM에 넣는 초기 프롬프트. 모두가 같은 항목 목록에서 출발합니다.
2. **리뷰의 기준**: 디자이너가 결과를 대조하는 의도의 기록.

이 이중 역할에는 반드시 인지해야 할 함정이 하나 있습니다.

> **순환성 문제:** 결과물은 PRD에서 파생됩니다. 따라서 "결과물이 PRD와 일치하는가?"는 거의 자동으로 '예'입니다. 즉 **적합성(conformance) 리뷰는 신호를 거의 주지 못합니다.**

리뷰어의 가치가 남는 지점은 두 곳뿐입니다.

- **PRD 자체의 판단**: 이 화면을 이 패턴/의도로 만드는 것이 옳은가.
- **텍스트 프롬프트가 완전히 규정할 수 없는 것**: 실제 데이터에서의 정보 위계, 각 상태가 실제로 '잘 읽히는지', 긴 한국어 문자열에서 레이아웃의 의미가 무너지지 않는지.

그래서 리뷰 루브릭(§9)은 **PRD 판단 수준**과 **실행 수준**을 명시적으로 분리하며, 반려(kick-back)는 생성보다 상류에 떨어지는 **PRD 수준 반려**가 될 수 있습니다. 이렇게 하지 않으면, 나쁜 PRD가 그 PRD에 완벽히 일치하는 UI를 만들어내고 그대로 리뷰를 통과합니다.

## 5. 워크플로우 (상태 머신)

제출물은 아래 상태를 거칩니다. 게이트를 건너뛸 수 없습니다.

```
PRD 작성 → 생성 → 자동 게이트 → 리뷰 → { 승인 | 표면 수정 | 반려 } → 머지
                      ▲                              │
                      └──────── (수정 시) ───────────┘
                                                     │
              (PRD 수준 반려는 생성 이전으로 되돌아감)  │
                                                     │
                    (반복되는 지적) ──────────────────┴──→ 하네스 업데이트
```

**PRD 작성 (작성자, 생성 전).** §7 템플릿으로 PRD를 작성합니다. 생성보다 먼저 써야 의도가 사후 합리화가 아니라 실제 입력이 됩니다. PRD 없이 생성 없음.

**생성 (작성자, 원하는 도구, 하네스 아래).** 코드를 만듭니다. `@enhans-jooyeon/synapse`에서 import합니다. 푸시 전 로컬 검사를 돌립니다.

**자동 게이트 (사람 없음).** CI가 §6 검사를 돌립니다. green이 되기 전까지 제출물은 **리뷰어에게 보이지 않습니다.** 이것이 병목 방지 장치이며, 선택 사항도 아니고 작성자가 무시할 수도 없습니다.

**리뷰 (리뷰어).** localhost 프리뷰(§8)를 직접 실행해 확인하고, 루브릭(§9)을 적용해 정확히 하나의 결과를 구조화된 피드백과 함께 반환합니다.

**머지 (머지 오너).** `승인`만 머지에 도달합니다. 디자인 승인은 **머지의 게이트**입니다 — 참고 의견이 아닙니다. (참고 의견 방식으로 하고 싶다면 지금 정해서 여기에 적으세요. 애매하게 두는 것이 나중에 갈등을 만듭니다.)

**하네스 업데이트 (하네스 오너, 정기).** 반복되는 지적은 variant/규칙/패턴이 됩니다. §10 참조.

### 5-1. 핸드오프와 브랜치 전략

작성자 → 리뷰어 → 머지의 코드 핸드오프는 **프로덕트 저장소의 main에서 분기한 브랜치**에서 진행합니다. 별도 저장소를 쓰지 않습니다. 이유는 이 하네스 전체를 관통하는 원칙과 같습니다 — 게이트, 프리뷰, 머지가 하나의 단일 소스를 공유해야 합니다.

- **별도 저장소는 지금까지 없애온 드리프트를 다시 불러옵니다.**
  - CI 게이트는 프로덕트 저장소에 설정되어 있습니다. 별도 저장소는 그 설정을 복제(드리프트)하거나, 진짜 게이트를 아예 돌리지 못합니다.
  - 화면 프리뷰에는 보통 app shell(라우팅, provider, mock)이 필요합니다. 별도 저장소에서는 이를 stub하거나(유지보수 부담) 통합 프리뷰가 불가능합니다.
  - 무엇보다 저장소 A → 프로덕트 저장소로 코드를 옮기는 **두 번째 핸드오프**가 생깁니다. "A에서 통과"가 "프로덕트에서 통과"와 같지 않으므로, 재통합할 때마다 새로운 부패 지점이 됩니다.

- **거버넌스 우려는 저장소 벽이 아니라 브랜치 보호로 해결합니다.** "PM이 AI 생성 코드를 프로덕션 저장소에 푸시해도 되는가"가 걱정이라면, 해법은 branch protection + CODEOWNERS + 디자인/엔지니어 필수 승인 + 머지 권한을 지연(머지 오너)으로 제한하는 것입니다. 이는 기존 모델(머지 오너 + 디자인 승인이 머지 게이트)을 강제하는 것일 뿐입니다. 저장소 벽은 위험을 없애지 못하고, 여전히 누군가 코드를 들여와야 하는 승격 단계로 미룰 뿐입니다.

- **구체적 형태:**
  - 짧게 유지되는 브랜치. 접두사(`ai/` 또는 `gen/`)로 생성 작업을 식별.
  - **Squash-merge** — 브랜치에서 AI 반복 작업이 아무리 지저분해도 main 히스토리는 깨끗하게 유지.
  - **브랜치별 프리뷰 배포**(이미 쓰는 Vercel)를 리뷰어의 무설정 localhost 대용으로. 클론 없이 클릭 한 번.
  - 필수 상태별 Storybook story를 브랜치 위 리뷰 단위로.

- **컴포넌트와 화면의 구분(중요):** 새 Synapse **컴포넌트**는 하네스/디자인 시스템 저장소에 들어갑니다 — 그 저장소는 올바르게 분리되어 있습니다. 기존 컴포넌트를 쓰는 생성된 **화면**은 프로덕트 저장소 브랜치로 갑니다. 생성물이 새 컴포넌트를 필요로 하면, 그것은 §9의 프로토콜대로 **반려 + 하네스 티켓**이지, 기능 브랜치 안에서 컴포넌트가 조용히 태어나는 게 아닙니다.

- **예외:** 프로덕트 저장소가 아직 없는 그린필드거나, 규제 대상 구매자가 생성 도구의 프로덕션 코드베이스 접근을 계약으로 금지하는 경우에 한해 별도 저장소를 고려합니다.

## 6. 자동 게이트 사양 (사람이 하지 않도록 CI가 강제하는 것)

스택(React + Radix + Tailwind + CVA + TS)에 맞춘 항목입니다. 각각 hard fail입니다.

- **임의 값 금지.** Tailwind가 임의 값(`[13px]`, `[#4f46e5]`)을 거부하도록 설정. lint 규칙이 JSX/CSS 내 raw hex/rgb/px 리터럴을 스캔. 토큰 사용을 강제.
- **컴포넌트 출처.** ESLint `no-restricted-syntax`로 Synapse 컴포넌트가 있는데 raw 엘리먼트(`<button>`, `<input>`, `<dialog>`…)를 쓴 경우 flag. 시스템에 있으면 시스템을 써야 함.
- **타입으로 variant 유효성.** TS 컴파일이 게이트. CVA variant가 타입으로 정의되어 있으므로 존재하지 않는 variant는 컴파일 실패. "규칙을 variant에 내장" 원칙이 빌드 타임에 공짜로 작동.
- **필수 상태 커버리지.** 작성자가 상태 매니페스트를 선언하고, CI는 선언된 각 상태에 대한 Storybook story 존재를 검증. 단일 상태(happy path만) 제출은 거부.
- **접근성 최소선.** `jsx-a11y` + axe를 CI에서 실행(대비, role, label, 타겟 크기). 토큰을 제대로 쓰면 대비는 이미 보장되며, 이 게이트는 오용을 잡음.
- **비주얼 스냅샷 생성.** Chromatic / Playwright로 모든 story를 캡처해, 리뷰어가 상태를 뒤지지 않고 한눈에 보게 함.

디자이너가 확인'했으면' 하지만 아직 자동화할 수 없는 검사가 있다면, 그것은 하네스 오너의 백로그이지 사람의 영구 업무가 아닙니다.

## 7. PRD 템플릿 (프롬프트 겸 리뷰 기준)

이 하나의 문서를 **LLM에 넣는 초기 프롬프트**로도 쓰고 **리뷰 기준**으로도 씁니다. 각 필드에 태그를 붙였습니다.

- **[생성 지시]** — 모델이 지시로 읽는 부분. 프롬프트 품질을 직접 좌우.
- **[공용]** — 생성·CI·리뷰에 모두 쓰임.
- **[판단 맥락]** — 사람(리뷰)을 위한 부분. 모델의 선택에도 도움.

> **작성용 템플릿은 별도 파일에 있습니다 → `템플릿-PRD.md`**
> 필드별 상세와 "프롬프트로 쓸 때의 순서" 가이드도 그 파일에 포함되어 있습니다. 이 가이드는 설명만 두고, 채워 넣는 실제 서식은 템플릿 파일 한 곳에서만 관리합니다(복제 = 드리프트 방지).

템플릿이 담는 항목: 컨텍스트(기능·사용자·진입점) / 생성 지시(하네스·참조 패턴·출력 형식) / 데이터·콘텐츠 예시(필수) / 필수 상태 매니페스트 / 제약·안티패턴 / 판단(의도적 일탈·반려한 대안) / 수용 기준 / 리뷰어에게 묻는 열린 질문.

## 8. 리뷰 패키지 / PR 템플릿 (localhost 실행 포함)

리뷰어가 **무엇을 볼지** 정확히 알려주고 **실행 가능한 산출물**을 넘기는 부분입니다.

> **작성용 템플릿은 별도 파일에 있습니다 → `템플릿-리뷰-패키지-PR.md`**
> `.github/PULL_REQUEST_TEMPLATE/ui_review.md` 로 저장하면 저장소에 기록되고 diff됩니다 — code-based end-to-end.

템플릿이 담는 항목: localhost 실행(명령어·주소·브랜치 프리뷰·상태별 경로) / 먼저 볼 것 / 상태 커버리지 표 / 브레이크포인트 확인 / 자동 게이트 상태 / 명시적 사인오프가 필요한 일탈.

## 9. 디자인 리뷰 루브릭 + 3가지 결과

**범위 규율:** 리뷰어는 **기계가 할 수 없는 것만** 봅니다. 토큰·대비·import·상태 존재 여부는 다시 검사하지 않습니다 — 게이트가 처리합니다. 그런 걸 발견했다면 게이트에 구멍이 있는 것이니, 업무로 흡수하지 말고 하네스에 이슈로 올리세요.

리뷰는 **두 수준**에서 이뤄집니다.

**A. PRD 판단 수준 (이걸 이렇게 만드는 게 옳은가)**
- 이 패턴이 이 일에 맞는가, 아니면 그럴듯한 오답인가.
- 의도/목표 자체가 타당한가.
- → 여기서 문제면 **PRD 수준 반려** (생성 이전으로).

**B. 실행 수준 (PRD가 완전히 규정할 수 없던 것 — 여기가 리뷰어의 진짜 가치)**
- 실제 데이터 밀도에서 정보 위계와 스캔 가능성.
- 각 상태가 '잘 읽히는가' — empty가 유용한가, error가 실행 가능한가, 긴 한국어에서 레이아웃의 **의미**가 무너지지 않는가(픽셀만이 아니라).
- 권한 variant가 단순 표시/숨김이 아니라 경험으로서 말이 되는가.
- 카피 품질과 톤(별도 카피 패스가 없다면).

### 3가지 결과 — **수정이 어디에 있는가**로 결정

| 결과 | 의미 | 누가 수정 | 머지 게이트 |
|---|---|---|---|
| **승인** | 의도 충족, 올바른 패턴, 모든 상태가 잘 읽힘. | — | ✅ 릴리스 |
| **표면 수정** | 접근은 옳고, 문제가 **기존 시스템 안**에 있으며 접근을 바꾸지 않음(토큰 교체, variant 선택, 기존 스케일 내 spacing, 카피). | 사소하면 리뷰어가 PR에 직접 수정 푸시; 아니면 *정확한* 주석(정확한 토큰/prop/값). 작성자가 다시 생각할 필요 없음. | ✅ 수정 후 |
| **반려** | 접근 또는 **시스템 자체**가 틀림: 의도 불일치, 잘못된 패턴, 누락된 상태, 혹은 Synapse가 아직 지원하지 않는 요구. PRD 수준 반려 포함. | 작성자에게 (재작업) — 또는 시스템이 빈 경우 하네스 결정 대기. **누락된 시스템 기능을 우회로 땜질하지 말 것.** | ❌ |

핵심 구분: **표면 수정 = 수정이 시스템 안에 있고 접근은 유지된다. 반려 = 접근이나 시스템이 문제다.** 커스텀 CSS나 일회성 컴포넌트로 "표면 수정"하고 싶어진다면, 그건 반려 + 하네스 티켓입니다.

**피드백 형식(모든 결과 공통):** 결과를 먼저 밝히고, 항목별로 *무엇을 → 왜 중요한지 → 구체적 변경 또는 필요한 결정*. "좀 더 살려주세요" 식 금지.

## 10. 피드백 루프 (하네스가 썩지 않도록)

**시스템 공백**(누락된 variant, 모호한 규칙, 없는 참조 패턴)을 지적한 모든 반려/표면 수정을 기록합니다. 정해진 주기(주 1회면 충분)마다 하네스 오너가 트리아지합니다.

- **2회 이상 반복 → 기계화.** 새 CVA variant, 새 lint 규칙, 또는 새 게이트. 이제 다시는 사람에게 오지 않음.
- **반복되는 판단 사안 → design.md에 참조 패턴 추가.** 다음 생성이 첫 시도에 맞게 나오도록.
- **모호한 규칙 → 독트린을 조임.**

이 루프가 없으면 리뷰 부하는 영원히 평평한 직선입니다. 있으면 감소합니다 — 반복 지적을 하네스가 흡수할수록 리뷰 대상 표면이 줄어듭니다. 리뷰 지적을 "다음에 무엇을 기계화할지"의 백로그로 다루세요.

## 11. 감시할 실패 모드 (분기별 점검)

- **리뷰가 만드는 것보다 느리다.** 사실이라면 게이트가 약한 것 — 기계화 가능한 일이 사람에게 새고 있음. 리뷰어가 아니라 게이트를 고치세요.
- **PRD가 생성 후에 작성되고 있다.** 증상: PRD가 결과물을 완벽히 묘사하고 반려한 대안이 없음. 명예 규칙이 아니라 워크플로우로 "생성 전 PRD"를 강제.
- **리뷰어가 스크린샷을 리뷰한다.** 증상: 상태/브레이크포인트 버그가 프로덕션으로 샘. localhost가 실행 불가하거나 실행되지 않는 것. 타협 불가 전제.
- **"표면 수정"이 비대해진다.** 결과 대부분이 표면 수정이면, 하네스가 너무 느슨하거나(게이트로 가야 함) 리뷰어가 반려를 회피하는 것. 경계선을 재조정.
- **같은 지적이 반복된다.** 피드백 루프가 깨진 것. 기계화 책임자가 있는데 안 하는 것.

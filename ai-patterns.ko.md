# Synapse AI 패턴

AgentOS에서 에이전트 동작에 대한 상호작용 규칙입니다. 이 패턴들은 AI 에이전트가 행동하고, 말하고, 제안하고, 콘텐츠를 생성하는 모든 표면을 규율합니다. `components.md`와 동일한 계약적 효력을 가지며, 여기에 정의된 상태와 구조는 닫힌 집합입니다.

기본 시스템 위에 적용됩니다. 아키타입과 밀도는 `patterns.md`에서, 컴포넌트는 `components.md`에서 가져옵니다(v1.2에서 추가된 AI 전용 컴포넌트 포함: AgentStep, ProposalCard, SourceChip, ProgressBar, CommandPalette).

---

## 1. 원칙 — 가시적 에이전시의 네 가지 속성

**충돌 방지(v6.11, 최종 형태 v6.19):** AI 테마는 완전히 슬레이트입니다 — 표면, 테두리, 마크, 솔리드 채움 모두 — 따라서 색상은 그 자체로 절대 AI를 신호하지 않습니다. **사각형 아바타가 주된 에이전시 표식**이며, `ai.fg` 슬레이트 마크와 배치가 이를 뒷받침합니다. `emphasis.*` 토큰은 AI 표면에 절대 나타나지 않습니다(`ai.*`를 사용하세요 — 같은 계열이지만 의미가 구별되며, 향후 자유롭게 분화할 수 있습니다). 대화형 AI 진입점은 `primary` Button입니다(v6.19; accent는 폐기됨).

모든 에이전트 표면은 에이전시를 **가시적**으로(사용자는 AI가 행동 중인지, 사람 또는 결정론적 시스템이 행동 중인지 항상 알 수 있음), **중단 가능**하게(어떤 생성이나 실행도 중지할 수 있음), **귀속 가능**하게(AI 출력은 그렇게 표시되며, 가능한 경우 출처와 함께), **검토 가능**하게(중대한 행동은 사람의 승인을 거침) 만들어야 합니다.

**두 가지 AI 표식.** AI 존재는 정확히 두 개의 장치로만 신호되며, 일관되게 사용되고 다른 어디에도 사용되지 않습니다:

1. 모든 행위자 행, 메시지, 또는 귀속 칩에 나타나는 **사각형 아바타**(둥근 사람 아바타 대비 반경 `sm`).
2. 에이전트 출력이나 에이전트 제안을 담은 표면의 **`ai.*` 토큰 계열**(`--sy-ai-surface`, `--sy-ai-border`, `--sy-ai-fg`), 그리고 화면의 단일 대화형 AI 진입점에만 사용되는 **`accent`(파란색) 채움**(v6.2.1 — 에이전트에게 묻기 / Composer 전송; 화면당 하나). 운영성 에이전트 행동(Run/Retry/Resume)은 일반 위계를 사용합니다 — "에이전트를 실행하는" 것은 결코 파란색을 얻지 못합니다.

AI 존재를 다른 어떤 방식으로도 절대 표시하지 마세요 — 반짝이는 그라디언트 금지, 보라색 금지, 표준 에이전트 글리프를 넘어서는 로봇 도상 금지. `ai.*` 토큰을 비-AI 표면에 절대 사용하지 말고, 두 계열이 관련되어 있더라도 `status.info-*`를 `ai.*`로 또는 그 반대로 절대 대체하지 마세요.

## 2. 스트리밍 출력

- 스트리밍된 텍스트는 **추가 전용**입니다: 렌더링된 마크다운은 점진적으로 향상될 수 있지만(제목이 완성되거나 목록 항목이 닫힘) 기존 줄은 절대 재배치되거나 튀지 않습니다. 가능한 한 블록 높이를 예약하세요.
- **스트리밍 커서**는 마지막 문자 뒤에 오는 2px × 1em `ai.fg` 세로 막대이며, 1초 간격으로 깜빡입니다. 시스템에서 깜빡임이 허용되는 유일한 요소입니다.
- 스트리밍 중: **Stop** 컨트롤(`secondary` 톤 Button, 사각형 정지 아이콘 + "Stop" / "중지" — v6.17.1: ghost가 맨 텍스트로 떠 있었음; 톤 채움이 유일한 중단 어포던스에 실체를 부여하며, Composer의 전송→중지 모프와도 일치하는데 이 또한 `secondary`임)이 스크롤 없이 보이고 도달 가능해야 합니다. 중지하면 부분 출력이 유지되고 `fg.tertiary` 캡션이 추가됩니다: "Stopped by you" / "사용자가 중지함".
- 스크롤 동작: 사용자가 하단에 있는 동안 하단에 고정; 사용자가 위로 스크롤하는 순간 잠금을 해제하고 "Jump to latest" 알약을 표시합니다(Toast 표면 — `bg.raised-2` + 테두리 + `shadow.overlay` — 하단 중앙).
- 생성 중에는 Composer를 절대 비활성화하지 마세요 — 대기열에 넣거나 중단하되 사용자를 잠그지 마세요.
- 속도: 자연스러운 토큰 도착 시점에 렌더링합니다. 스트리밍되지 않은(이미 완성된) 콘텐츠에는 인위적인 타자기 효과 금지 — 즉시 렌더링, 가짜 지연 절대 없음.

## 3. 작업 상태 — AgentStep

"요청"과 "답변" 사이의 모든 에이전트 활동은 **AgentStep** 행으로 렌더링됩니다(`components.md` 참조). 닫힌 상태 집합:

| 상태 | 표시자 | 텍스트 색상 |
|---|---|---|
| `pending` | 12px `border.strong` 빈 원 | `fg.tertiary` |
| `running` | 12px Spinner | `fg.secondary` |
| `success` | 12px `status.success` 체크 | `fg.tertiary` |
| `failed` | 12px `status.danger` ✕ | `fg.secondary` + 후행 Retry ghost 버튼 |
| `skipped` | 12px `border.strong` 대시 | `fg.disabled` |

- 스텝은 13px 행이며, 각각 한 줄입니다: 동사 우선 요약("Fetched 328 tickets" / "문의 데이터 328건 조회됨") + 선택적 소요 시간(`fg.tertiary`, tabular-nums).
- **접기 규칙:** 실행 중에는 최근 3개 스텝까지만 표시하고, 완료 시 단일 요약 행으로 접습니다 — "5 steps · 12s" / "5단계 · 12초" — 펼칠 수 있습니다. 펼쳐진 스텝 상세(도구 페이로드)는 `.sy-code-block`으로 렌더링되며 기본적으로 접혀 있습니다.
- 스텝 위계는 평면이거나 한 단계 깊이입니다. 절대 더 깊게 중첩하지 마세요 — 대신 에이전트의 보고 방식을 재구성하세요.

## 4. 도구 호출

- 도구 호출은 요약에 도구 이름을 mono로 담은 AgentStep입니다: `slack.post_message` — 사람이 읽을 수 있는 요약이 먼저, mono 식별자가 두 번째("Posted summary to #ops · `slack.post_message`").
- 입력과 출력은 기본적으로 접혀 있으며, 펼치면 복사 버튼이 있는 `.sy-code-block`이 표시됩니다. 비밀 정보는 기본적으로 가립니다(`••••`, 표시하려면 명시적 클릭과 권한 필요).
- **모델 선택(v6.42):** 에이전트 피커 옆의 대화별 Composer 컨트롤입니다(v6.41의 피커 내 배치를 뒤집음). 선택된 에이전트 구성의 기본값을 따르며, 에이전트를 전환하면 해당 에이전트의 기본값으로 초기화됩니다. 조직 정책이 읽기 전용으로 잠글 수 있습니다. 모델 선택은 권한, 도구 접근, 승인 규칙을 절대 변경하지 않습니다.
- **사용자 도구 제어(v6.41):** Composer의 도구 팝오버는 이 대화에서 에이전트가 사용할 수 있는 기능/커넥터를 토글합니다(에이전트 구성의 기본값을 따름). 에이전트가 필요로 하는 도구를 비활성화하면 → 에이전트가 이름을 지정해 요청하며, 조용히 실패하거나 조용히 다시 활성화하지 않습니다. 도구를 활성화하는 것은 승인을 절대 우회하지 않습니다: 다음 규칙은 토글과 무관하게 유지됩니다.
- **외부 부작용**이 있는 도구 호출(전송, 게시, 구매, 삭제, 서드파티 시스템에 쓰기)은 사용자가 해당 특정 도구+범위에 대해 상시 승인을 부여한 경우가 아니라면 ProposalCard(§5)를 반드시 거쳐야 하며, 부여된 경우 스텝에 "pre-approved" Badge(`neutral`)가 표시됩니다.

## 5. 사람 개입 승인 — ProposalCard

에이전트는 중대한 행동을 절대 조용히 수행하지 않습니다. 에이전트는 제안하고, 사람이 결정합니다.

**구조**(`components.md` 참조, v6.20 트레이): 테두리 없는 `ai.surface` 객체, 반경 `lg`, 그림자 없음 — Composer 트레이 언어; 페이로드 표면(diff, 미리보기)은 `bg.page`로 열립니다. 헤더 행: 사각형 에이전트 아바타가 먼저(에이전시 표식) + 에이전트 이름 + "proposes" / "제안"(13 medium, `ai.fg`), 아래에 `ai.border` 헤어라인(v6.21.1 — 단일 톤 트레이; 투톤 밴드는 뒤집혔음). 본문: 행동과 그 범위에 대한 한 문장 요약, 그다음 페이로드 — diff, 메시지 미리보기, 영향받는 레코드 목록. 푸터: **Approve**(`primary` Button — 승인은 에이전트 작업에 대한 사람의 결정이지 AI CTA가 아님; v6.2) + **Reject**(`secondary`) + 선택적 **Edit**(`ghost`).

**규칙:**
- 시간 초과로 자동 승인 절대 금지. 승인에 카운트다운 타이머 없음. 응답 부재 = 무행동.
- 파괴적 제안은 Approve를 `danger` Button으로 바꾸고 결과를 건수와 명사로 반드시 명시해야 합니다("Deletes 14 runs permanently" / "실행 기록 14개가 영구 삭제됩니다").
- 일괄 제안(에이전트가 N개의 유사한 행동을 제안)은 Approve all / Review individually를 허용합니다 — 절대 approve-all을 기본 포커스 컨트롤로 두지 않습니다.
- 해결된 ProposalCard는 단일 귀속 행으로 접힙니다: 체크/✕ 아이콘 + "Approved by June · 14:02" / "June 승인 · 14:02". 결정 이력은 영구적입니다 — 해결된 제안은 대화 기록에서 절대 삭제되지 않습니다.

**Diff 렌더링** 제안 내부: 추가된 줄은 `status.success-bg`에, 제거된 줄은 `status.danger-bg`에, 코드/구성은 mono, 산문 diff는 sans; 변경된 줄 내에서 단어 단위 강조. 절대 빨강/초록 텍스트만 쓰지 마세요 — 배경 틴트에 +/− 거터 마커를 더합니다(색각 이상 안전).

## 6. 출처 — SourceChip

- 검색 가능한 소스에서 파생된 모든 에이전트 주장은 인라인 **SourceChip**을 지녀야 합니다(SHOULD): 번호가 매겨진 칩 `[1]`(18px 높이, `bg.sunken`, 반경 `xs`, mono 11)이 뒷받침하는 문장 뒤에 배치됩니다. 호버/클릭 시 Popover가 열립니다: 소스 제목, 출처(파비콘 또는 커넥터 아이콘), 타임스탬프, 소스 열기 링크.
- 출처 푸터는 메시지의 모든 인용을 나열합니다(13px, `fg.secondary`).
- 검색 가능한 소스가 없고 검증 가능성이 낮은 주장은 메시지당 한 번 `neutral` Badge로 표시됩니다: "Model knowledge" / "모델 지식" — 절대 인용을 위조하지 말고, 에이전트 자신을 절대 인용하지 마세요.
- 에이전트가 생성한 *문서*(채팅뿐 아니라)의 숫자, 인용문, 명명된 사실은 동일한 규칙을 따릅니다. AgentOS가 에이전트가 작성한 보고서를 렌더링하면 칩이 함께 옵니다.
- **출처 행(v6.36):** 메시지 아래에서 출처는 출처 `micro-label` 아래에 컴팩트한 ContextCard(인덱스 + 아이콘 타일 + 이름)로 렌더링됩니다 — 절대 평문 각주 줄로 렌더링하지 않습니다. 카드↔마커 호버 연동이 매핑을 시각화합니다. 인라인 마커는 문장당 3개 이하로 유지합니다.
- **마커 형태(v6.37):** emphasis 틴트 위에 맨 숫자가 있는 원형 18px 칩 — 인용은 조용한 강조입니다; 대괄호와 mono는 폐기됩니다(주석이 아니라 코드로 읽힘).

## 7. 불확실성

- 3단계 어휘만 사용: 기저 시스템이 실제로 보정된 경우가 아니면 원시 백분율을 쓰지 않습니다. Badge로 렌더링: high = `success` "Verified" / "확인됨"(소스와 대조해 확인된 경우에만), medium = `neutral` "Likely" / "추정", low = `warning` "Unverified" / "미확인".
- 저신뢰 출력은 승인된 템플릿을 사용해 카피에서 추가로 완곡 표현합니다("Based on limited data, …" / "제한된 데이터 기준으로는 …"). 완곡 표현은 텍스트에, 불확실성 등급은 Badge에 삽니다 — 둘 다에 완곡 표현을 중복하지 마세요.
- 불확실성을 불투명도, 이탤릭(어차피 금지), 또는 장식용 물음표로 절대 표현하지 마세요.

## 8. 중단과 취소

- 모든 생성: Stop 컨트롤(§2). 모든 장기 실행: Cancel(실행 툴바의 `ghost`).
- 외부 부작용이 없는 실행 취소: 즉시, 확인 없음. 부작용이 이미 실행된 경우: Modal이 확인 전에 무엇이 일어났고 무엇이 일어나지 않았는지 명시합니다("2 of 5 messages already sent. Cancel the remaining 3?").
- 취소된 실행의 상태는 `warning` Badge "Cancelled" / "취소됨" — `failed`와 구별됩니다.

## 9. 귀속

- 에이전트가 생성한 산출물(문서, 테이블 행, 구성)은 귀속 행을 지닙니다: 사각형 아바타 + 에이전트 이름 + 타임스탬프, 12px `fg.tertiary`.
- 사람이 에이전트 출력을 편집하면 귀속이 "AI-drafted · edited by June" / "AI 초안 · June 편집"으로 바뀝니다 — AI 출처는 절대 조용히 사라지지 않으며, 사람의 편집은 절대 AI 출력으로 제시되지 않습니다.
- 혼합 목록(사람과 에이전트 행위자)에서 모든 행은 그 행위자의 아바타를 표시합니다; 형태만으로(둥근 것 대 사각형) 작성자를 훑어보기에 충분해야 합니다.

## 10. 실패와 복구

- 실패한 실행이나 스텝은 오류 EmptyState 유형을 인라인으로 사용합니다: 무엇이 실패했는지, 왜인지 한 문장으로(스택 트레이스가 아닌 원인), Retry 행동. 스택 트레이스와 원시 오류는 펼칠 수 있는 `.sy-code-block` 뒤에 삽니다.
- 카피는 사용자를 절대 탓하지 않고, 고통을 의인화하지 않으며("I'm so sorry!!"), 원인에 대한 확신을 절대 가장하지 않습니다. 중립적이고, 사실적이며, 실행 가능하게: "The Slack connector rejected the request (rate limit). Retrying usually resolves this." / "Slack 연동이 요청을 거부했습니다(요청 한도 초과). 다시 시도하면 대부분 해결됩니다."
- 자동 재시도가 2회 실패한 후에는 재시도를 중단하고, 수동 경로 대안이 있으면 그것과 함께 사용자에게 에스컬레이션합니다.
- 부분 성공은 부분으로 보고됩니다: "Processed 312 of 328"에 16건의 실패로 가는 링크 — 절대 성공으로 반올림하지 않습니다.

## 11. 장기 작업 — ProgressBar

- 길이를 아는 작업: 확정 ProgressBar(`components.md` 참조) + "N of M" 캡션. 알 수 없는 경우: 불확정 바 + 현재 AgentStep을 레이블로. 10초를 초과하는 작업에는 절대 맨 스피너를 쓰지 마세요.
- 사용자가 다른 곳으로 이동하면 완료는 Toast로 알립니다(View 행동 포함); 실패는 Toast danger 변형으로 알립니다. 진행 중인 실행은 실행 목록에서 항상 찾을 수 있습니다 — 백그라운드 작업은 절대 방치되지 않습니다.
- 시간 추정은 이력에서 파생된 경우에만("usually ~4 min" / "평균 4분 소요") — 절대 지어내지 않습니다.

## 12. 에이전트 마크다운 렌더링(v5.1)

에이전트 출력은 마크다운입니다; 고정된 렌더링 규칙이 없으면 모든 메시지가 저마다의 타이포그래피를 즉흥적으로 만듭니다. 매핑은 닫혀 있습니다:

- **제목은 강등됩니다:** 에이전트 `#`는 `heading-md`로, `##`는 `heading-sm`으로, 더 깊은 단계는 semibold `body`로 렌더링됩니다. 에이전트 텍스트는 절대 `heading-xl/lg`를 생성하지 않습니다 — 페이지 위계는 메시지가 아니라 페이지에 속합니다.
- **본문** = `body` 스타일; 목록은 `space-4` 항목 간격을 가지며, 한 단계 중첩까지 렌더링되고 더 깊은 것은 평면화됩니다.
- **코드:** 펜스 → 코드 블록 처리(`bg.sunken`, `code` 스타일, 복사 버튼, 언어 칩, 최대 높이 400px + 펼치기); 인라인 → `sy-code-inline`.
- **링크** = `fg.link`이며 앱 외부 대상에는 외부 마크; 맨 URL은 자동 링크되고 중간이 잘립니다.
- **테이블**은 맨(테두리 없는) 조밀 테이블로 렌더링됩니다 — 헤어라인 헤더 규칙, `label` 헤더; 메시지 열보다 넓으면 → 가로 스크롤, 절대 재배치 안 함.
- **인용문:** 2px `border.strong` 왼쪽 규칙 + `fg.secondary`; 한 단계, 더 깊은 것은 평면화.
- **이미지:** 사용자/워크스페이스 첨부만 인라인으로 렌더링(최대 너비 100%, 반경 `md`, `caption` 캡션); 원격 URL은 링크로 렌더링되고 절대 가져오지 않습니다 — 출처와 안전. **사람 메시지(v6.28–28.1):** 첨부는 버블 텍스트 위에 고정 순서로 쌓입니다 — 문서 ContextCard가 먼저, 그다음 이미지(버블 정렬, 반경 `md` + 헤어라인, 최대 높이 240; 나란히 2개, 3개 이상은 2열 그리드, 절대 MediaGroup 팬 아님 — 그것은 생성 미디어 전용), 그다음 텍스트.
- **작업 목록** → 읽기 전용 Checkbox; **hr** → 전체 폭 `border.subtle`; 강조는 시스템 규칙을 따름(bold = 600; 이탤릭은 foundations §2.3.2에 따라 600으로 정규화).
- **스트리밍 안전:** 블록 요소(테이블, 펜스)는 해당 블록이 닫힐 때 렌더링됩니다 — 부분 테이블은 절대 스타일 없이 번쩍이지 않습니다; 텍스트는 §2에 따라 스트리밍됩니다.

## 13. AI 진입점으로서의 CommandPalette

팔레트(⌘K / Ctrl+K, `components.md` 참조)는 보편적 진입 표면입니다: 내비게이션, 행동, 그리고 일치하는 결과가 없을 때 마지막 항목으로 나오는 "에이전트에게 묻기" 탈출구("Ask agent: '{query}'"에 accent 처리). 이는 크롬 전체에 accent 버튼을 흩뿌리지 않고 모든 화면에 AI 어포던스를 부여합니다 — 그 희소성을 보존하세요.

## 14. 추론 공개(v5.2)

제품이 에이전트의 작업/추론 텍스트를 노출할 때, 그것은 답변 콘텐츠가 아니라 **공개 행**으로 렌더링됩니다:

- 기본적으로 접힘: 셰브론 + "Reasoning" / "추론 과정"(`label`, `fg.tertiary`) + 소요 시간. 펼치면 `bg.surface` 위에 `body-sm` `fg.secondary`로 텍스트가 드러나며, 에이전트 마크다운 규칙(§12)으로 렌더링되지만 제한됩니다: 제목 없음, 이미지 없음.
- 추론은 시각적으로 답변에 종속됩니다 — 절대 `fg.primary`를 쓰지 않고, 절대 SourceChip을 지니지 않으며(인용은 *답변*의 주장에 속함), 복사/재생성에서 제외됩니다(ResponseToolbar는 답변에만 작용).
- 펼침 상태는 대화별 사용자별로 유지됩니다; 자동 펼침은 금지됩니다.
- 추론은 정책에 따라 가려질 수 있습니다; 가려진 섹션은 그것을 명확히 말합니다("Reasoning not available for this response" / "이 응답의 추론 과정은 제공되지 않습니다") — 절대 빈 채로 렌더링하지 않습니다.

## 15. 가드레일 공지(v5.2)

거부나 차단된 행동은 *정책 결과*이지 오류가 아닙니다 — 오류 스타일을 입어서는 안 됩니다:

- `bg.sunken` 위에 방패 아이콘과 `fg.secondary` 텍스트가 있는 인라인 공지로 렌더링합니다(중립 Banner 처리). 절대 `status.danger` 금지 — 빨강은 사용자에게 정책이 고장이라고 가르칩니다.
- 카피는 훈계하지 않고 범주와 나아갈 길을 명명합니다: "This request can't be completed under your workspace's data policy. An admin can review the policy in Settings." / "워크스페이스 데이터 정책에 따라 처리할 수 없는 요청입니다. 정책은 관리자가 설정에서 확인할 수 있습니다."
- 부분 차단(일부 소스가 답변에서 제외됨)은 `caption` 줄 + SourceChip 끊김 상태로 표시됩니다 — 답변은 렌더링되고, 제외는 가시화됩니다.
- 정책 차단을 기술적 실패로 절대 위장하지 말고, 그 반대도 절대 금지합니다.

## 16. 사람에게 인계(v5.2)

에이전트가 사람에게 에스컬레이션할 때(또는 사람이 인수할 때):

- 인계는 대화 행으로 렌더링됩니다: 전달 아이콘 + "Handed off to {name}" / "{name}님에게 전달됨"(`caption`, `fg.tertiary`)에 담당자의 둥근 Avatar — 그 행부터 행위자 형태가 뒤집히고 뒤집힌 채로 유지됩니다. 아바타 형태 시스템(둥근 것=사람, 사각형=에이전트)이 상태를 전달합니다; 추가 크롬 없음.
- 대기 상태는 상태 Badge입니다: "Needs review" / "확인 필요"(`warning` subtle)이며, 목록에 나타나는 곳 어디서든 실행/작업에 붙습니다.
- 에이전트에게 *되돌리는* 것은 명시적 행동입니다 — 영역의 위계에 따라 `primary` 또는 `secondary`("Resume agent" / "에이전트 계속하기"); 이것은 운영성이지 대화형 진입이 아니므로 절대 `accent` 금지. 에이전트는 사람이 인수한 작업을 절대 조용히 되찾지 않습니다.
- 혼합 활동 피드에서 인계는 일급 이벤트이며, 절대 인접 행에서 추론되지 않습니다.

## 17. 사용량과 한도(v5.2)

- 할당량 표시는 ProgressBar 사용량 관할입니다(`components.md`): 확정, tabular-nums 값, ≥80%에서 warning 채움, 100%에서 danger와 플랜 한도 Banner("Workspace is out of agent credits — runs are queued." / "워크스페이스 크레딧이 소진되어 실행이 대기 중입니다.").
- 실행에 표시되는 비용은 실제값이며, 로케일 형식입니다(content.md §6); 추정은 추정으로 레이블되고 이력에서 파생되며 절대 지어내지 않습니다(§11).
- 한도 접근 상태는 우아하게 저하됩니다: 에이전트는 요청을 계속 받아 대기열에 넣습니다; Composer는 할당량으로 절대 잠기지 않습니다 — Banner가 상태를 전달합니다.

## 18. 선택과 답장(v6.12)

에이전트 메시지 내에서 선택된 텍스트는 일급 대화 객체입니다.

**답장 알약:** 에이전트 메시지에서 텍스트를 선택하면 단일 부동 알약이 떠오릅니다("Reply" / "답장", 답장 아이콘) — `bg.raised-2`, `border.overlay`, `shadow.overlay`, 알약 반경, `z.dropdown`, 표준 등장. 하나의 알약, 옵션 툴바 없음; 선택 해제나 esc 시 사라집니다.
**인용 강조:** 선택된 구절은 인용된 동안 `ai.surface` 채움 + 삽입된 `ai.border` 헤어라인을 취합니다.
**ComposerQuote:** 답장을 선택하면 Composer의 텍스트영역 위에 인용 바가 삽입됩니다 — `ai.surface` 채움, `ai.fg` 텍스트, 답장 아이콘, 한 줄 말줄임, × 제거. 반경 `xs`(동심: composer 반경 16 − 12 패딩 = 4). 전송당 최대 하나의 인용; 인용하면 기존 인용을 대체합니다. 전송된 메시지는 사용자 텍스트 위에 인용을 렌더링합니다.

**선택 행동(v6.47, 닫힌 집합):** 부동 알약은 답장 · 설명 · 재생성을 지닙니다 — 그리고 거버넌스 없이는 그 외에 아무것도 없습니다(행동은 행동 용어집에서 옵니다). 설명은 인용된 후속 질문을 구성하며("이 부분을 설명해 줘" + 선택의 ComposerQuote); 설명은 스레드에 일반 에이전트 턴으로 도착합니다. 선택 행동은 절대 원본 메시지를 제자리에서 변경하지 않습니다 — 스레드 이력은 추가 전용입니다(출처 법칙). 재작성 유형 행동은 사용자 자신의 초안(§24)에만 적용되며, 절대 에이전트 출력에는 적용되지 않습니다.

## 19. 후속 질문 패널(v6.12)

제안 칩(Chip `suggestion`, 최대 3개)이 수동적 기본값으로 유지됩니다. Composer가 포커스되고 후속 질문이 존재할 때, 그 위에 앵커된 패널이 열릴 수 있습니다: **glass 재질**(v6.23 — foundations §5의 스크림 게이팅에 대한 유일한 앵커 예외: 패널은 미니 팔레트처럼 스레드 콘텐츠 위에 떠 있고, 열려 있는 동안 정적이며, esc/타이핑 시 사라짐; 투명도 감소 시 불투명으로 폴백)에 `border.overlay`, `shadow.overlay`, 반경 `md`, 6px 패딩, 반경 `xs`의 32px 행; 행은 12px 후속 화살표로 시작하며, 키캡으로 된 전체 폭 키보드 헤더 행(↑↓ 이동 · ↵ 선택 · esc 닫기). **배치(v6.23.1–2):** 패널은 **Composer의 상단 가장자리 위 8px**에 절대 앵커되며(부동 레이어는 앵커에서 분리됨 — 밀착 접촉은 입력의 일부로 읽힘; 앵커된 메뉴는 4px를 쓰지만, 패널의 더 큰 질량이 8을 얻음) 마지막 스레드 메시지를 **오버레이**합니다 — 절대 콘텐츠를 아래로 밀지 않습니다(열 때 레이아웃 이동은 금지되며, glass 재질은 뒤에 콘텐츠가 없으면 무의미합니다). 최대 4행; 선택하면 텍스트가 Composer에 삽입됩니다(절대 자동 전송 안 함). 칩과 패널은 절대 동시에 표시되지 않습니다.

**칩 정직성(v6.47, 채택된 법칙):** 제안 칩의 보이는 레이블은 그것이 전송하는 쿼리 그 자체입니다 — 절대 더 길거나 다른 숨겨진 프롬프트가 아닙니다. 실제 쿼리가 칩이 표시할 수 있는 것보다 더 많은 단어를 필요로 하면, 칩은 전송 대신 편집을 위해 Composer에 삽입합니다.

## 20. 답변 구조와 명명된 작업 상태(v6.13)

**제목이 있는 답변 섹션:** ~4문단보다 긴 에이전트 답변이나 다단계 실행으로 생성된 답변은 답변 헤더로 시작할 수 있습니다: 제목(`heading-sm`, 실행의 명시된 목표에서) + 총 소요 시간 Badge(`neutral` subtle, tabular 숫자) + 접기 셰브론(ghost 아이콘 버튼). 접으면 제목 + 소요 시간 유지; 펼침 상태는 대화 기록에 유지됩니다. 답변당 하나의 헤더 — 절대 문단당 하나가 아님.
**명명된 작업 줄:** 다단계 생성 중, 스텝 위의 상태 줄이 스텝 계획에서 현재 활동을 명명합니다("보고서 초안 생성 중…" / "Drafting the report…"), `body-sm` `fg.secondary`에 `pulse` 불투명도 애니메이션(스켈레톤 전례 — 그라디언트/샤이머는 여전히 금지). 스텝 이름이 존재할 때마다 맨 스피너를 대체합니다; 중지 컨트롤은 §2에 따라 인접하게 유지됩니다. 이 줄은 완료 시 답변 헤더의 제목으로 해결됩니다.

## 21. 생성 미디어(v6.16)

에이전트가 생성한 미디어는 MediaGroup 팬으로 렌더링됩니다(components.md 참조). 미디어 전용 답변은 팬 옆에 `media` ResponseToolbar 레일을 지닙니다; 텍스트와 미디어를 혼합하는 답변은 메시지 수준 ResponseToolbar만 사용합니다 — 절대 둘 다 아님(v6.17.7). 지배 원칙: **유희성은 에이전트의 출력에 살고, 절대 크롬에 살지 않습니다** — 시스템에서 유일한 유희 장치인 회전은 에이전트가 만든 미디어에 국한됩니다. 출처는 여전히 적용됩니다: 생성 미디어는 표준 귀속 행을 지니며, 이미지 콘텐츠는 텍스트와 동일한 정직성 규칙을 따릅니다(주장에 대한 가짜 사진 "증거" 금지).

## 22. 결과 변형과 부분 재생성(v6.25)

**변형 페이저:** 에이전트 답변은 자신의 여러 변형을 담을 수 있습니다. 답변 헤더(§20)는 오른쪽 정렬 페이저를 얻습니다: `caption` tabular "2/2" 주위의 ‹ › ghost 아이콘 버튼(sm). 최신 답변에서 재생성하면 변형 N+1이 생성됩니다 — 절대 파괴하지 않습니다; 전환은 비파괴적입니다; 각 변형은 자신의 출처와 귀속을 유지합니다. 최대 5개 변형, 그다음 재생성은 고정되지 않은 가장 오래된 것을 대체합니다.
**부분 재생성:** §18 선택 알약은 두 개의 행동을 지닙니다 — "답장"과 "재생성" — 헤어라인으로 구분됩니다. 재생성은 선택된 구절만 제자리에서 재생성합니다: 새 텍스트가 일시적 `emphasis.surface` 번쩍임과 Undo Toast(경량 되돌리기 관례)와 함께 도착합니다. 절대 세 번째 알약 행동 없음; 범위는 선택이며, 절대 그 주위의 문단이 아닙니다.

## 23. 프롬프트 템플릿과 플레이스홀더(v6.25)

**라이브러리:** 프롬프트는 저장되고("템플릿으로 저장"은 전송된 메시지의 ⋯ 오버플로에서) 두 가지 방식으로 소환됩니다 — Composer의 `/` 범위가 템플릿 그룹을 얻고(전문가 빠른 삽입), **북마크 아이콘 버튼이 Template Library Modal(v6.30–31 — 메뉴에서 업그레이드됨: 제목만 있는 행은 "이 템플릿이 내 프롬프트에 무엇을 하는가?"에 답할 수 없음; 라이브러리는 커밋 전에 콘텐츠를 보여줌)을 엽니다.** Modal 구조(불투명 `bg.raised`, **760 — 브라우즈 라이브러리 너비, 명명된 Modal 예외(v6.34): 폼은 640에서 제한, 데이터 검토 Drawer는 800, 라이브러리는 그 사이**, **투페인**): 헤더(제목 + 닫기, **세로 16 / 좌우 24 — 세로는 헤어라인에 대칭; 좌우는 페인의 24 콘텐츠 열과 일치**, v6.34.2)가 헤어라인 위에 · 왼쪽 목록 열(260, **`bg.surface` 채움 — 레이어드 페인 읽기**: 전체 폭 검색 행 · **범위 SegmentedControl(전체 / 내 템플릿 / 팀, v6.34)** · **즐겨찾기 그룹이 항상 먼저**(별 토글, v6.34.1 — 핀/고정됨이었음) — 즐겨찾기가 볼륨 답입니다: 라이브러리 크기와 무관하게 작업 세트가 상단에 유지됨 — 그다음 범위별 `micro-label` 그룹 헤더, **호버 시 드러나는 별 토글**(20px 컴팩트, `aria-pressed`; 유휴 = 선; **활성 = `emphasis.fg-soft`에서 채워짐(v6.35 — slate.500/400, 가장 밝은 법적 값: 3:1 비텍스트 하한보다 한 단계 위, 게이트 검증됨; 채움이 질량을 지녀 텍스트가 할 수 없는 방식으로 마크에 가벼움을 부여함) — 유일하게 승인된 활성 시 채움 아이콘: 즐겨찾기 토글의 임무는 상태를 알리는 것이며, 선+색상만으로는 비활성으로 읽힘.** 썸네일과 다른 모든 아이콘은 선 전용을 유지합니다; 즐겨찾기된 행은 별을 계속 보이게 유지)이 있는 테두리 없는 32px 행, 선택 = `bg.selected`; "새 템플릿 만들기"가 전체 폭 헤어라인 위 하단에 고정, 행 주위 균일한 8px 패딩(v6.32.1)) · 오른쪽 미리보기 페인(균일한 24 패딩): **오른쪽 상단에 별 토글이 있는 아이브로 록업(24px, 행 상태를 미러링)** — 그룹을 `heading-sm` 이름 위 `micro-label` 아이브로로(아이브로가 소유권을 지님; 배지 불필요), 한 줄 설명, **인용문으로 된 클로즈 미리보기**(2px `border.strong` 왼쪽 규칙 — 시스템의 인용 언어, 슬롯은 `emphasis.surface`로 강조), 그리고 오른쪽 하단에 삽입(`primary`). 설명 아래에 `caption` 메타 줄(소유자 · 마지막 편집 — v6.32), 클로즈 아래에 `caption` 슬롯 요약("입력 항목 N개 · 삽입 후 →로 이동" — v6.32.2, → 동작을 가르치는 기능적 텍스처); 페인 푸터는 왼쪽에 키캡 힌트(↵ 삽입 · esc 닫기, `micro` + `.sy-kbd`)를, 오른쪽에 삽입(`primary`)을 짝지으며, ↵는 선택된 템플릿을 삽입합니다. 열 내부 규칙(검색 행, 고정 푸터)은 분할선 법칙에 따라 열 가장자리까지 확장됩니다. **규모에서(v6.33):** modal 본문은 높이를 고정하고(420) 행 영역이 독립적으로 스크롤됩니다 — 스크롤바는 스크롤 컨테이너 법칙에 따라 열 가장자리에 — 고정 그룹 헤더와 함께(표면 채움이라 행이 아래로 미끄러짐); 검색 필터는 제목에 작동해 비워진 그룹을 숨기며, 컴팩트한 결과 없음 줄('…에 대한 결과가 없습니다'); ↑↓는 보이는 행을 통해 선택을 이동하고(뷰로 스크롤), ↵는 삽입합니다. 행을 선택하면 미리보기; 삽입 또는 ↵는 슬롯 칩과 함께 삽입하고 닫습니다. 콘텐츠는 페인에 한 번 나타납니다 — 절대 행마다 반복되지 않음(박스 안의 박스는 와이어프레임 공식의 사촌).
**플레이스홀더:** 템플릿은 Composer 텍스트에 인라인 슬롯 칩으로 렌더링되는 명명된 슬롯을 지닙니다 — `emphasis.surface` 채움, `fg.secondary`, 반경 `xs`, "[기간]" 레이블. →는 슬롯 사이를 이동합니다(ghost 완성과 일관; 절대 Tab 아님 — IME). 채워지지 않은 슬롯으로 전송하면 슬롯을 명명하는 캡션 오류로 차단됩니다(빈 전송 규칙을 확장; 유일하게 다른 승인된 전송 차단).
**이중 언어 하드 규칙:** 템플릿은 슬롯이 있는 완전한 로케일별 문장으로 작성됩니다 — 슬롯에는 절대 한국어 조사가 붙지 않습니다(content.md §4); 문장에 조사가 필요하면 슬롯이 조사 없이 자리하도록 템플릿을 다시 씁니다.

## 24. 작성 코치(v6.25)

**품질 힌트:** Composer가 포커스된 동안, 그 아래에 구체적 개선을 명명하는 하나의 `caption` `fg.tertiary` 줄이 나타날 수 있습니다("기간을 지정하면 더 정확한 결과를 얻습니다"). 잔소리 방지 규칙은 절대적입니다: 최대 하나의 힌트만 표시, 절대 전송을 차단하거나 지연하지 않음, 전송이나 편집 시 사라짐, 같은 초안에 대해 해제 후 절대 재등장 안 함, 절대 warning/danger 색상 안 씀 — 코칭은 제안이지 관문이 아닙니다.
**프롬프트 다듬기(v6.27 확장 — 보조 편집):** 펜 라인 `ghost` 아이콘 버튼(맥락적, v6.44: 초안이 비어 있지 않은 동안에만 입력의 오른쪽 상단에 떠 있음)이 **프리셋 다듬기**의 작은 메뉴를 엽니다 — 닫힌 집합: 전체 다듬기(일반; v6.47.3에서 프롬프트 다듬기에서 재레이블됨 — 그 문자열이 메뉴의 micro-label 헤더가 되었고, 행은 절대 자신의 헤더를 중복해선 안 됨), 더 자세히, 더 간결하게, 기간·범위 구체화, 형식 지정. 메뉴 구조는 피커 계열과 일치합니다(v6.47.3): `micro-label` 헤더 + 행 + 프리셋 그룹 앞 구분선; 프리셋 목록은 이 스펙의 일부이며 거버넌스로만 확장됩니다(자유형 "재작성 스타일"은 금지 — 그것은 입력으로의 용어집 표류). 선택된 재작성은 Composer 텍스트를 Undo Toast와 함께 대체합니다(경량 되돌리기); 빈 초안에서는 버튼이 렌더링되지 않습니다(v6.44 — 가시성이 비활성화를 대체: 맥락적 어포던스는 비활성으로 앉아 있기보다 적용 가능할 때 나타남). 절대 자동 아님; 재작성된 텍스트는 사용자가 편집할 것 — 입력 내부에 출처 표시 없음(작성자는 사용자로 유지).
**미채택:** 프롬프트 텍스트의 생성형 인라인 자동완성. Ghost 완성은 닫힌 용어집을 유지합니다 — 사용자 입력 내부의 모델 출력은 출처가 표시될 수 없는 곳에서 작성자를 흐립니다.

## 25. 스레드(v6.25)

**이력:** 대화는 Sidebar에 `micro-label` 시간 그룹(오늘 / 이전 — NotificationCenter와 동일한 그룹화 언어) 아래 표준 내비게이션 항목으로 나열됩니다, 제목 한 줄 말줄임, 활성 = `bg.selected`. 최대 2개 그룹 표시, 그다음 "모두 보기".
**새 대화:** "새 대화" — 대화 그룹 상단의 ghost 버튼(플러스 아이콘 + 텍스트); ⌘K도 항상 제공합니다.
**임시 대화:** 에이전트 피커 푸터 영역의 Switch 행("임시 대화" + 캡션 "기록에 저장되지 않습니다"); 활성 동안 Console 헤더에 `neutral` Badge "임시" 표시 — 상태는 설정만 되는 게 아니라 가시화되어야 합니다.

## 26. 음성 입력(v6.26 — 유지관리자 결정; 이전의 "보류" 결정을 대체)

**범위: 받아쓰기이지 음성 메시지가 아님.** 음성은 Composer에서 편집 가능한 텍스트가 됩니다 — 사용자가 검토하고 전송합니다. 오디오는 절대 스레드에 게시되지 않습니다: 에이전트 대화는 텍스트로 유지되어 출처, 검색, 인용 답장이 성립합니다. 작성자는 사용자로 유지됩니다(§24 논리 — 대화 기록은 사용자 자신의 음성).

**트리거:** Composer 푸터의 후행 측, 전송 바로 왼쪽의 마이크 `ghost` 아이콘 버튼(v6.43 — 선행 그룹에서 이전됨: 받아쓰기는 곧 전송될 메시지를 채움). 절대 ⋯ 오버플로 내부 안 됨 — 음성 입력은 한 번의 탭 거리에 있어야 합니다.
**녹음 상태:** Composer 트레이가 제자리에서 모프됩니다(절대 별도 오버레이 아님): Cancel(`ghost` Button, "취소") · 맥동하는 8px `status.danger-bg-solid` 점 + tabular 타이머("0:04") · 컴팩트 레벨 미터(≤5 막대, 2px 폭, `fg.tertiary`, transform-scaleY 애니메이션만 — 모션 법칙; 모션 감소 시 정적) · 일시정지 `ghost` 아이콘 버튼 · 전송 위치의 **`primary` 아이콘 전용 원(체크)**으로서의 확인 — 전송↔중지와 동일한 모프 언어. Esc는 취소; ↵는 확인.
**확인 후:** 받아쓰는 동안 작업 줄("받아쓰는 중…", pulse), 그다음 대화 기록이 캐럿에 삽입됩니다 — 기존 초안에 추가되며, 절대 자동 전송 안 함.
**접근성:** 녹음 상태 변경은 `aria-live`로 알림; 째깍이는 타이머는 `aria-hidden`(매초가 아니라 시작/중지 시 알림); 마이크 버튼은 `aria-label`에 상태를 반영.
**금지:** 받아쓰기 종료 시 자동 전송; 스레드의 오디오 산출물; 녹음 바 외부의 파형 장식; 가시적 danger 점 표시자 없는 녹음.


## 27. 시작 제안(v6.47)

**제로 상태 전용.** 빈 대화(새 세션, 턴 없음)는 Composer 위에 3–4개의 시작 칩을 표시할 수 있습니다 — `tag-sugg` 구조, §19 제안과 동일한 객체(시스템 전체에 하나의 제안 언어). 시작 제안을 선택하면 편집을 위해 Composer에 삽입됩니다 — 절대 자동 전송 안 함(§19 칩 법칙 적용).

- 시작 제안은 워크스페이스 도메인에서 완전하고 잘 구성된 프롬프트를 모델링해야 합니다 — 절대 "무엇이든 물어보세요"가 아님(프롬프트 구조를 가르치지 않는 시작 제안은 장식). **레이블 ≠ 삽입(v6.47.4, 데모 검증):** 칩 레이블은 짧은 작업 핸들(2–4단어)입니다; 선택하면 전체 예시 프롬프트가 삽입됩니다(범위, 형식, 제약이 명시됨). 레이블은 핸들이고, 삽입이 교훈입니다. 칩 정직성(§19)은 보존되는데 시작 제안은 편집을 위해 삽입하기 때문입니다 — 절대 전송하지 않음.
- 첫 세션 후 역할과 최근성으로 개인화(예: 사용자가 떠난 곳에서 재개); 세션 하나를 넘어서 일반 시작 제안을 돌리는 것은 안티패턴입니다.
- 해제 가능(행 끝의 ×); 해제는 사용자별로 유지됩니다. 선택된 에이전트가 제공할 수 없는 기능을 절대 광고하지 마세요.
- 첫 턴에 사라짐; §19 후속 질문이 인계받습니다. 둘은 절대 함께 렌더링되지 않습니다.

## 28. 첨부 인텔리전스(v6.47)

첨부 후, 칩/타일은 자문 캡션을 얻을 수 있습니다: 콘텐츠 판단이 아니라 추출된 형태("CSV · 328행 · 7/6–7/12 접수 문의") — `caption` 스타일, `fg.tertiary`, 자동 요약으로 표시. 법칙: **자문 전용** — 절대 전송을 차단하지 않고, 절대 첨부를 수정하지 않으며, 절대 전송에 필수가 아님; 대기 상태는 `pulse` 불투명도 애니메이션 사용(명명된 작업 상태 계열, §20); 분석 실패는 조용히 평범한 칩으로 폴백(선택적 편의에 오류 표면 없음); 모든 생성 텍스트처럼 에이전트에 귀속. 목적: 사용자가 실행을 소비하기 전에 올바른 파일이 들어갔는지 확인.

## 29. 일괄 입력 처리(v6.47)

한 번에 제출된 여러 동종 입력(파일, 레코드, 쿼리)은 메가 프롬프트가 아니라 **대기열**로 실행됩니다:

- 항목별 행: mono 파일명 + 진행 바(R-progress 구조) + 상태(완료 / n% / 대기 / 실패).
- 개별 제어: 행별 일시정지와 취소(컴팩트 20px 아이콘 버튼, v6.27.2 계열); 실패 행은 개별적으로 재시도 — 한 번의 실패가 절대 일괄을 중단하지 않음.
- 일괄 푸터: 집계 줄("2/4 완료 · 예상 3분") + 결과 내보내기; 결과는 Table(레시피 R15)에 착지, 항목당 한 행.
- 일괄 내부의 외부 부작용은 여전히 ProposalCard(§5)를 거칩니다 — 일괄 처리는 절대 승인을 우회하지 않음; 사전 승인된 도구 예산은 항목별로 적용됩니다.
- 일괄 실행 중 Composer는 사용 가능한 상태로 유지됩니다(절대 대기열에서 차단 안 함).

## 30. 예측 텍스트(v6.47 — 슬래시 명령 ghost 완성을 일반화)

전체 프롬프트 연속은 캐럿 뒤에 ghost 텍스트로 렌더링될 수 있습니다: `fg.placeholder`, 최대 한 줄, 평문 전용.

- **→는 수락. Tab은 절대 수락 안 함**(IME 법칙, 슬래시 완성 및 템플릿 슬롯과 동일한 결정).
- 한글 조합이 활성인 동안 완전히 억제됨(compositionstart→compositionend) — 조합 중인 음절 내부의 ghost 텍스트는 읽을 수 없음.
- 계속 타이핑 = 해제; 명시적 해제 컨트롤 없음. Ghost 텍스트는 전송이 제출하는 것을 절대 바꾸지 않음 — 수락된 텍스트만 실제입니다.
- 소스: 닫힘 — 행동 용어집, 사용자 자신의 최근 프롬프트, 템플릿 제목. 자유형 모델 생성 연속은 거버넌스 제안을 필요로 함(지연 + 잘못된 ghost 비용: 나쁜 ghost 텍스트는 없는 것보다 나쁨).

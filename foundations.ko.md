# Synapse foundations

토큰 레이어에 대한 근거와 사용 규칙. 토큰 값은 `tokens/synapse.tokens.json`(정본)과 `tokens/synapse.css`(생성됨)에 있습니다. 이 문서는 *언제*와 *왜*를 설명합니다 — JSON만 읽는 에이전트는 기술적으로 유효하지만 판단이 부실한 UI를 만듭니다. 이 파일이 그 판단을 제공합니다.

규칙 키워드: **MUST** / **NEVER**는 기계로 시행 가능한 제약입니다. **SHOULD**는 강한 기본값을 의미하며, 벗어나려면 이유를 명시해야 합니다. **MAY**는 재량입니다.

---

## 1. 색상

### 1.1 성격

Synapse는 뉴트럴, 블랙 키(black-key) 시스템입니다. 인터페이스는 거의 전적으로 그레이 램프로 구성됩니다. 색상은 오직 의미에만 지출하는 희소 자원입니다. 의도한 인상은 절제된 엔지니어링입니다 — 소비자용 앱보다 정밀 계측기에 가깝습니다.

- 주 액션 색상은 **검정**(다크 모드에서는 흰색)입니다. 파랑이 아닙니다.
- **포인트 컬러(v6.51): `--sy-brand-point` / accent `#0A84FF`.** 브랜딩 팀 팔레트에서 가져온, 시스템의 유일한 선명한 액센트입니다. 검정은 브랜드/액션 색상으로 유지되며, 포인트 컬러는 오직 두 관할에만 사용됩니다: (1) **브랜드 아이덴티티 요소** — 워크스페이스 모노그램 타일, 문서 허브/프리뷰 브랜드 마크, 허용된 Artific 브랜드 히어로 모먼트, 로고/마케팅 표면; (2) **AI 강조** — 대화형 AI CTA(`accent` Button: Ask agent / Composer send, 화면당 최대 1) 및 **진행 중인 AI 실행 상태 표시**(실시간 에이전트 비콘, 진행 중 `ai` ProgressBar — `ai.solid` 경유). 일반 크롬, 운영성 에이전트 액션(Run/Retry/Resume은 primary/검정 유지), 또는 슬레이트 AI 표면 위의 텍스트 틴트로는 쓰지 않습니다(v6.19의 '색이 뒤섞인다'는 반려가 여전히 유효 — 슬레이트는 AI *표면*으로 남고, 포인트 파랑은 그 위의 별개 솔리드 액센트이며 결코 슬레이트 위 파랑이 아닙니다).
- **기능적 파랑은 별개로 유지됩니다.** `fg.link` / 포커스 링 / `status.info`는 더 차분한 인디고(`#3155C6`)를 유지합니다 — 역할(실용적 신호)도 색조도 포인트 컬러와 달라, 기능과 브랜드/AI-강조가 결코 충돌하지 않습니다. 포인트 파랑의 흰색 레이블은 §8 솔리드 레이블 ≥3:1 + 600 굵기 정책을 따릅니다(흰색 위 `#0A84FF` = 3.63:1).
- 슬레이트(`emphasis.*`, `ai.*` 표면, 그리고 슬레이트 값을 갖는 선택 상태)는 조용한 강조 패밀리입니다 — 선명한 색 없이 중요도를 표시하기 위한, 살짝 차가운 틴트입니다. 두 번째 뉴트럴이 아닙니다: 기본 표면, 보더, 텍스트에는 절대 쓰지 않으며, 오직 닫힌 직무 목록을 통해서만 사용합니다(테이블 강조 열/셀, 타임라인/캘린더의 now-marker, 선택 상태, AI 표면 — v6.18. hero-stat-card 직무는 v6.40에서 제거됨). 충돌 방지: 틴트만으로는 결코 AI를 신호하지 않습니다 — 각진(squared) 아바타가 주 행위자 마커이고, `ai.fg` 슬레이트 마크가 이를 보조합니다. `emphasis.*` 토큰 자체는 AI 표면에 절대 나타나지 않습니다(`ai.*`를 사용, 값은 같고 시맨틱은 다름).
- 파랑(`border.focus`, `fg.link`, status.info 패밀리)은 *기능적*이며 브랜드가 아닙니다: 링크, 비입력 컨트롤의 키보드 포커스 링, 정보성 상태. AI *표면*은 슬레이트로 유지되지만(ai.surface/border/fg, v6.19 이후), AI *강조*는 이제 포인트 컬러를 취합니다(v6.51 — 위 포인트 컬러 항목 참조); `action.accent-*`가 그 포인트 컬러이며, 대화형 AI CTA와 진행 중 실행 상태 표시를 위해 되살아났습니다. 이 기능적 파랑 역할(링크/포커스/정보)은 결코 겹치지 않는, 더 차분한 별개 색조입니다. 파랑이 존재하는 이유는 순수한 모노크롬으로는 상호작용성과 포커스를 접근성 있게 신호할 수 없기 때문입니다. 예외(v6.10): 입력 표면은 뉴트럴 `border.focus-input` 테두리로 포커스합니다 — 텍스트 필드는 클릭해 타이핑할 때마다 포커스를 보이며, 캐럿 + 필 변화 + 보더 교체가 복합 지시자를 형성하므로 accent가 필요 없습니다.
- 상태 색상은 소리치지 않고 뉴트럴 필드 안에 자리 잡지만, 탁하지 않고 유채색입니다(v3.4–3.5에서 재보정): 텍스트 토큰은 자신의 틴트 위에서 AA 대비가 허용하는 만큼 채도를 지니고, 모든 솔리드 필은 중간 톤에 흰색 `fg.on-solid` 텍스트를 씁니다 — success/warning 솔리드는 문서화된 솔리드 레이블 정책(§8)에 따라 약 3.5:1로 동작합니다.

### 1.2 규칙

- 에이전트는 시맨틱 토큰(`--sy-bg-*`, `--sy-fg-*`, `--sy-border-*`, `--sy-action-*`, `--sy-status-*`)을 MUST 사용합니다. 원시 hex 값과 프리미티브 램프 참조는 생성된 UI에서 NEVER 허용되지 않습니다.
- 한 화면 영역에는 솔리드 필 accent 요소가 최대 하나(예: primary 버튼 하나)만 SHOULD 있습니다. 나머지는 모두 아웃라인, ghost, 텍스트 스타일을 사용합니다.
- 상태 색상은 상태 전용입니다. `--sy-status-danger`를 장식으로, `--sy-status-success`를 범용 "초록 accent"로 NEVER 사용하지 마십시오.
- 솔리드 레드 필(`danger-bg-solid`)은 파괴적 확인 버튼에만 예약됩니다.
- 데이터 시각화는 `--sy-viz-1` … `--sy-viz-8`을 순서대로, 건너뛰지 않고 배정해 MUST 사용합니다. 차트가 상태(pass/fail, healthy/degraded)를 인코딩하면 viz 토큰 대신 상태 토큰을 사용합니다.
- 유채색 상태 배경 위 텍스트는 페어링된 상태 전경 토큰을 MUST 사용하며, `fg.primary`는 절대 쓰지 않습니다.
- 모든 텍스트/배경 쌍은 WCAG 2.1 AA(본문 4.5:1, 큰 텍스트 ≥18px 세미볼드 또는 ≥24px는 3:1)를 MUST 충족합니다. 토큰 파일의 시맨틱 쌍은 사전 검증되어 있습니다. 새로운 조합은 금지되며, 이로써 검증이 불필요해집니다.

### 1.3 레이어링 모델

모든 표면은 다섯 레벨 중 정확히 하나에 놓입니다. 다크 모드에서는 배경이 깊이를 표현하고, 라이트 모드에서는 보더와 섀도가 깊이를 표현합니다 — 레벨 토큰이 둘 다 자동으로 처리합니다.

| 레벨 | 토큰 | 라이트 | 다크 | 용도 |
|---|---|---|---|---|
| L0 | `bg.page` | white | black | 페이지 자체. |
| L1 | `bg.surface` | gray-50 | gray-950 | 그룹핑 영역: 사이드바, 테이블 헤더, 툴바, 플랫 카드. |
| L2 | `bg.raised` | white + border | gray-900 | 카드, 모달, 드로어, 메뉴 — 자체 경계를 가진 모든 것. |
| L3 | `bg.raised-2` | white + border + shadow | gray-800 | 레이어 위의 레이어: 모달에서 연 팝오버, 중첩 패널, 드래그 중인 항목. |
| well | `bg.sunken` | gray-100 | gray-950 | 오목한 콘텐츠: 코드 블록, 입력 웰, 스켈레톤 필, 드롭존. |

- 떠 있는 레이어는 자신의 동작에 맞는 섀도 티어(`raised`/`overlay`/`modal`)를 추가로 취합니다 — 섀도는 *떠 있음*을, 레벨은 *쌓임*을 전달합니다.
- 중간 그레이를 새로 만들지 말고, 레벨을 건너뛰지 마십시오(모달에서 연 팝오버는 L2가 아니라 L3입니다 — 다크 모드에서는 두 개의 L2 표면이 시각적으로 합쳐집니다).
- 상호작용 틴트(`bg.hover`, `bg.active`, `bg.selected`, `bg.selected-hover`)는 레벨 *안에서* 적용됩니다. 이들은 레벨이 아닙니다.

### 1.4 토큰 선택 맵 — 어떤 경우에 어떤 토큰

사용 사례별 전체 시맨틱 어휘입니다. 어떤 경우가 여기 없고 명확히 맞는 토큰도 없다면, 그것은 제안 대상입니다 — 원시 값이 아닙니다.

| 경우 | 토큰 |
|---|---|
| 페이지 / 그룹핑 영역 / 카드 / 레이어 위의 레이어 / 오목한 웰 | `bg.page` / `bg.surface` / `bg.raised` / `bg.raised-2` / `bg.sunken` |
| 행 또는 항목 hover · pressed · selected · selected 상태의 hover | `bg.hover` · `bg.active` · `bg.selected` · `bg.selected-hover` |
| disabled 컨트롤 필 · disabled 텍스트 · disabled 보더 | `bg.disabled` · `fg.disabled` · `border.subtle` |
| 기본 텍스트 · 보조 텍스트 · 타임스탬프/힌트 · placeholder 및 혼합 값 마커 | `fg.primary` · `fg.secondary` · `fg.tertiary` · `fg.placeholder` |
| 링크 · `bg.inverse` 스트립 위의 링크 | `fg.link` · `fg.link-inverse` |
| 컴포넌트 경계 · 내부 디바이더 · 보더가 있는 요소의 hover | `border.default` · `border.subtle` · `border.strong` |
| 포커스 링 · 선택된 카드 아웃라인 · 유효하지 않은 필드 보더 | `border.focus` · `border.selected` · `border.error` |
| primary 버튼 rest/hover/pressed + 텍스트 | `action.primary-bg` / `-hover` / `-active` + `action.primary-fg` |
| AI 액션 버튼 rest/hover/pressed + 텍스트 | `action.accent-bg` / `-hover` / `-active` + `action.accent-fg` |
| 파괴적 확인 필 + 텍스트 | `status.danger-bg-solid` + `action.danger-fg` |
| 상태 텍스트/아이콘 · 상태 틴트 · 상태 솔리드 필 · 솔리드 위 텍스트 | `status.*` · `status.*-bg` · `status.*-bg-solid` · `fg.on-solid` |
| `bg.inverse` 표면 위의 상태/링크 색상 | `status.*-inverse` · `fg.link-inverse` |
| 에이전트 표면 · 에이전트 보더 · 에이전트 텍스트/마크 | `ai.surface` · `ai.border` · `ai.fg` |
| 조용한 강조 (hero stat, 테이블 강조 열, now-marker) | `emphasis.surface` · `emphasis.border` · `emphasis.fg` |
| 범주형 차트 시리즈 (고정 순서) | `viz-1` … `viz-8` |
| 모달 백드롭 · 인버스 강조 블록 | `bg.scrim` · `bg.inverse` + `fg.inverse` |
| 아이덴티티 틴트 (아바타, 카테고리 Badge/Chip) | 결정론적 `viz-n` at 20% + 매칭 `viz-n` 텍스트 |

## 2. 타이포그래피

### 2.1 타입 패밀리

세 개의 패밀리, 각각 닫힌 관할을 가집니다. 관할 밖에서 패밀리를 사용하는 것은 계약 위반이며, 원시 hex 값과 같은 중대도입니다.

| 패밀리 | 토큰 | 관할 |
|---|---|---|
| **Pretendard Variable** | `font.family.sans` | 모든 UI 텍스트, KO와 EN 동일. 기본값. 다른 두 패밀리에 명시적으로 부여되지 않은 모든 것. |
| **Artific** (Power Type Foundry) | `font.family.display` | 스타일화된 브랜드 모먼트에만 — 아래 참조. |
| **JetBrains Mono** (D2Coding Hangul 폴백) | `font.family.mono` | 코드와 데이터 식별자 — 아래 참조. |

두 문자 체계에 하나의 sans를 쓰면 언어 간 베이스라인 드리프트, 웨이트 불일치, 폴백 플래싱이 없어집니다.

**디스플레이 패밀리(Artific) 규칙.** Artific은 브랜드 모먼트에서 시각적 임팩트를 만들기 위해 존재합니다. 희소성이 이것을 브랜딩으로 읽히게 만듭니다.

- Display 크기(30/40, 36/48)에서만, 그리고 오직 다음에서만 허용됩니다: Guided 아키타입 hero(온보딩, 로그인, 최초 실행), empty-state hero 타이틀, 마케팅 인접 표면. 본문, 컨트롤, 테이블, 내비게이션, 또는 밀집 영역에는 NEVER.
- 화면당 디스플레이 패밀리 요소는 최대 하나.
- 웨이트는 600/700만, `.sy-display`를 통해서. 이 패밀리의 Thin–Medium 웨이트와 모든 oblique 스타일은 시스템에서 제외됩니다.
- **Artific은 영어 전용입니다(v6.48 — 하드 룰).** 이 패밀리에는 한글 글리프가 없으므로 Artific은 **라틴/영어 카피만** 스타일링할 수 있습니다. `.sy-display` / `sy-type-display*` 요소는 한글을 MUST NOT 포함합니다 — 이것은 감내할 폴백 상황이 아니라 설계로 회피해야 할 콘텐츠 제약입니다(SY016이 시행). 한 문자열 안에서 패밀리를 혼합해 스타일링하려 하지 말고, 시스템 제안 없이 다른 스타일화된 한국어 서체로 대체하지 마십시오.
- **Artific 헤더의 KO 로케일화는 영어 카피를 그대로 유지합니다(v6.48).** Artific이 짧은 브랜드 타이틀(몇 단어짜리 Guided/empty-state/HOME hero)을 스타일링할 때, 그 문자열은 **로케일화 가능한 UI 카피가 아니라 브랜드 모먼트**입니다: KO 로케일은 *같은 영어 단어*를 여전히 Artific으로 렌더링합니다 — 번역하지 않으며 Pretendard-한국어 렌더링으로 폴백하지도 않습니다. 예: HOME 인사말은 두 로케일 모두에서 "Good morning"으로 유지됩니다. 이것은 "모든 UI 텍스트는 로케일화된다"는 규칙(content §8)에 대한 의도적 예외이며, Artific 브랜드 타이틀에만 한정됩니다 — 다른 모든 문자열은 정상적으로 로케일화됩니다. 어떤 표면이 진정으로 한국어 디스플레이 텍스트를 필요로 한다면, 그것은 Artific을 얻지 못합니다: 디스플레이 크기의 Pretendard 볼드(정상적인 비브랜드 헤딩)를 사용하면, 콘텐츠 언어에 따라 브랜드 서체가 바뀌는 일이 없습니다.
- 라틴 디스플레이 텍스트는 -1% letter-spacing을 쓸 수 있습니다(`.sy-display`에 내장). 한글은 letter-spacing을 절대 받지 않습니다.
- 라이선싱: 상용 패밀리, `assets/fonts/`에 셀프 호스팅된 woff2(600/700만). 공개 CDN에 없음. 출시 전 Enhans의 라이선스가 앱 임베딩을 포함하는지 확인하십시오.

**Mono 패밀리 규칙.** Mono는 "기계적으로 유의미한 텍스트 — 그대로 정확히 복사하라"를 신호합니다.

- 사용처: 코드 블록과 인라인 코드(`.sy-code-block`, `.sy-code-inline`), ID/해시/API 키, 로그와 터미널 출력, 파일 경로, 키보드 힌트(`.sy-kbd`).
- 사용하지 말 것: 숫자 테이블 열(sans + `tabular-nums`를 사용 — mono는 식별자용이지 수량용이 아님), 날짜, 통화, 또는 어떤 산문에도.
- Mono 텍스트는 해시/ID에 한해서만 무(無)절단 규칙에서 면제되며, 이들은 중간을 생략(`a3f8…c92e`)해 MUST 절단하고 클릭 시 복사를 제공합니다.

### 2.2 타이포그래피 스타일 — 닫힌 타이포그래피 어휘

타이포그래피는 임의의 크기/웨이트 조합이 아니라 이름 붙은 **스타일**을 통해 설정합니다. 각 스타일은 완전한 레시피(패밀리 + 크기/line-height + 웨이트)이며 `.sy-type-*` 클래스로 렌더링되고 토큰 파일의 `semantic.type`에 정의됩니다. 텍스트 요소가 어떤 스타일에도 맞지 않으면, 그것은 커스텀 조합의 핑계가 아니라 제안 대상입니다.

| 스타일 | 스펙 | 관할 |
|---|---|---|
| `display-xl` | display 44/56 bold | 마케팅 인접 hero(sign-in, 런치 모먼트). 플로당 최대 하나. |
| `display` | display 36/48 semibold | Guided 아키타입 hero, empty-state hero. |
| `display-sm` | display 30/40 semibold | 컴팩트 브랜드 모먼트, Guided 스텝 타이틀. |
| `heading-xl` | sans 24/34 bold, 라틴 트래킹 −1% | 페이지 타이틀. 페이지당 하나. |
| `heading-lg` | sans 18/27 semibold, 라틴 트래킹 −1% | 섹션, 카드, 모달, 드로어 타이틀. |
| `heading-md` | sans 16/24 semibold | 서브섹션, empty-state 타이틀, 설정 그룹 타이틀. |
| `heading-sm` | sans 14/22 semibold | 카드 안의 그룹 타이틀, 밀집 패널 헤더. |
| `body-lg` | sans 16/24 regular | 긴 글 읽기: 에이전트 보고서, 문서, 온보딩 산문. 최대 폭 680px. |
| `body` | sans 14/22 regular | 어디서나 기본 텍스트. |
| `body-sm` | sans 13/20 regular (고정) | 메뉴 항목, 토스트 텍스트, 크기가 변하면 안 되는 메타 문장. |
| `label` | sans 13/20 medium | 폼 레이블, 테이블 헤더, 버튼. |
| `label-sm` | sans 12/18 medium | 컴팩트 레이블, 보조 테이블 헤더. |
| `caption` | sans 12/18 regular | 헬퍼 텍스트, 각주, 출처 표시 행. |
| `micro` | sans 11/16 semibold | Badge, kbd 힌트. 문장은 NEVER. 최저 크기가 강화된 웨이트를 지님: 11px에서 500은 흐려지고(특히 한글), 700은 카운터를 막음. |
| `micro-label` | sans 11/16 semibold, 라틴 트래킹 +2% | Eyebrow/그룹 레이블: 사이드바 그룹, 팔레트 그룹, 카드 eyebrow, 차트 축 레이블(v6.5). 트래킹이 ALL-CAPS가 할 일을 대신함 — 대문자는 여전히 금지(§2.3). 한글은 절대 트래킹하지 않음. |
| `code` / `code-sm` | mono 13/20 · 12/18 | 코드 블록, 로그 / 인라인 코드, 셀 안의 ID. |

규칙:

- 타입은 단일 스케일입니다 — `heading-sm`/`body`/`label`은 고정 크기입니다(v6.52). Synapse는 전반에서 단일 사이즈 스케일을 사용합니다(v6.53 — focus/dense 모드 제거).
- 한 표면 내 위계에는 최소 2단계 스타일 간격 또는 색상 변화(`fg.primary` vs `fg.secondary`)가 필요합니다 — 인접 스타일만으로는(16 vs 14) 실수처럼 읽힙니다.
- 400/500/600/700 외의 웨이트를 NEVER 사용하지 마십시오. 본문 길이 텍스트에 600+을 NEVER 사용하지 마십시오. 스타일이 있는 곳에 원시 `font-size`를 NEVER 사용하지 마십시오.
- 통계 스타일은 항상 `tabular-nums`를 지니며, 문장이 아니라 숫자와 단위를 위한 것입니다.

### 2.3 이중 언어(KO/EN) 규칙 — 협상 불가

이 규칙들은 같은 문자열의 한국어 버전과 영어 버전이 체계적으로 다르기 때문에 존재합니다: 한국어는 UI 레이블에서 흔히 10–20% 더 넓고, 글리프가 더 크며, 이탤릭이 없고, 절단 동작이 다릅니다.

1. **고정 폭 텍스트 컨테이너 금지.** 버튼, 탭, 배지, 메뉴 항목은 콘텐츠 + 패딩에서 크기를 MUST 결정합니다. 최소 폭은 허용됩니다. 번역 가능한 텍스트를 담은 어떤 것에도 고정 폭은 NEVER 허용되지 않습니다.
2. **이탤릭은 절대 금지.** 한글에는 이탤릭 형태가 없고, 합성 oblique는 판독 불가입니다. 강조 = 웨이트 600 또는 보조 텍스트 대비 `fg.primary`. CSS 레이어가 `<em>/<i>`를 강제로 정규화합니다.
3. **line-height는 최저 기준입니다.** 스케일의 페어링된 line-height는 한글의 어센트/디센트를 수용합니다. 절대 좁히지 마십시오. 본문 텍스트에 커스텀 `line-height < 1.4`는 금지됩니다.
4. **한국어 줄바꿈:** 모든 한국어 텍스트에 `word-break: keep-all; overflow-wrap: break-word;`(CSS 레이어에서 `:lang(ko)`로 적용). 문서 또는 영역 루트에 `lang="ko"` / `lang="en"`으로 언어를 표시하십시오.
5. **레이아웃은 텍스트 폭 +25%에서 견뎌야 합니다.** 레이블을 담은 컴포넌트를 디자인할 때 문자열 길이 125%에서 레이아웃을 검증하십시오. 깨진다면 번역이 아니라 디자인이 잘못된 것입니다.
6. **절단:** 한 줄 말줄임 절단은 테이블 셀과 리스트 행에서만 허용되며, 절단된 모든 문자열은 복원 가능해야 MUST(툴팁 또는 상세 뷰) 합니다. 버튼, 폼 레이블, 오류 메시지, 헤딩은 NEVER 절단하지 마십시오.
7. **ALL-CAPS 스타일 금지.** `text-transform: uppercase`는 한글에 아무 효과가 없으므로 혼합 언어 UI가 일관되지 않게 렌더링됩니다. 위계는 대신 크기/웨이트/색상에서 나옵니다.
8. **이미지/아이콘 속 텍스트 지양.** 렌더링되는 모든 텍스트는 로케일화될 수 있도록 라이브 텍스트여야 합니다.
9. **줄 밸런싱(v5.0.2).** 디스플레이 스타일, `heading-xl/lg`, 그리고 hero/empty-state 설명 문단은 `text-wrap: balance`를 취합니다 — 짧은 여러 줄 텍스트가 한 긴 줄에 고아 조각이 남는 대신 고른 줄로 나뉩니다. 이는 `keep-all`이 단어를 보존하지만 가운데 정렬된 짧은 텍스트에서 극심한 rag를 만드는 한국어에서 두 배로 중요합니다. 긴 본문 텍스트는 면제입니다(밸런싱은 ≤ 약 4줄용). hero 문단에는 `.sy-balance` 유틸리티를 사용하십시오.

## 3. 스페이싱

4px 기본 그리드. 스케일은 `--sy-space-*`에 열거되어 있습니다. 스케일에 없는 값(예: 10px, 18px, 25px)은 NEVER 허용되지 않습니다. 승인된 예외 하나: 1px 보더와 짝지은 헤어라인 오프셋으로서의 ±1px(예: 탭의 활성 밑줄이 컨테이너 룰과 겹치는 경우).

- 관련 항목: 4–8. 그룹핑된 컨트롤: 8–12. 그룹 사이: 16–24. 섹션 사이: `--sy-section-gap`. 페이지 패딩: `--sy-page-padding`.
- **디바이더는 컨테이너를 가장자리에서 가장자리까지 걸칩니다.** 패딩이 있는 표면(메뉴, 카드, 리스트) 안에서 디바이더는 패딩 안으로 확장됩니다(컨테이너 패딩과 같은 음수 마진) — 가장자리에 못 미치고 멈추는 디바이더는 실수처럼 읽힙니다. 디바이더가 가장자리에 *닿으면 안 된다면*, 디바이더 대신 스페이싱을 사용하십시오: 모든 간격은 분명히 의도적이거나 아예 없거나 둘 중 하나입니다.
- 레이아웃 수준 스페이싱(페이지 패딩, 섹션 간격, 카드 패딩, 스택 간격)은 원시 스케일 값이 아니라 `--sy-*` 스페이싱 토큰을 MUST 사용합니다.
- 수직 리듬 경험칙: 헤딩 *위* 간격은 그 아래 간격의 약 2배여야 합니다.

## 4. 사이징

Synapse는 **단일 사이즈 스케일**을 사용합니다(v6.53 — 기존 `focus`/`dense` 밀도 모드는 불필요한 복잡성으로 제거됨). 컨트롤, 스페이싱, 타입은 어디서나 하나의 스케일이며, `data-density`도, 영역별 모드도 없습니다.

- 유일한 예외는 **Table**로, 데이터가 스캔 가능하도록 기본적으로 컴팩트하게 동작합니다(`--sy-table-row` 36, `--sy-table-cell-x` 12) — 이는 Table의 일반 수치이지 모드가 아닙니다.
- 레이아웃 폭은 밀도가 아니라 아키타입의 선택입니다: 읽기 아키타입은 콘텐츠를 `--sy-content-max`(760px)로 제한하고, 워크벤치/데이터 영역은 유동입니다(`patterns.md` §1).
- 모든 크기/스페이싱은 원시 값이 아니라 `--sy-*` 컨트롤·레이아웃 토큰에서 MUST 가져옵니다.

## 5. 엘리베이션과 보더

Synapse는 **보더 우선(borders-first)**입니다: 흐름 안의 위계는 섀도가 아니라 1px 보더와 배경 단계로 그립니다.

- `border.subtle` — 컴포넌트 내부의 디바이더(테이블 행, 리스트 구분선).
- `border.default` — 컴포넌트 경계(카드, 입력, 테이블 프레임). v4.0 이후 헤어라인처럼 조용함.
- `border.strong` — 상호작용 가능한 보더 요소의 hover 상태.
- `border.overlay` — 떠 있는 레이어(메뉴, 팝오버, 토스트, 툴팁): **라이트 모드에서는 투명** — 부드러운 섀도가 가장자리를 전달함 — 다크 모드에서는 보이며, 여기서는 섀도가 검정 위에서 죽습니다.

**스크롤바(v6.46):** 법으로 조용함 — 얇고(8px), 트랙 없고, 보더 없음: 썸(thumb)만, 투명 트랙 위 `border.strong`, 래디우스 `full`, hover 시 `fg.tertiary`. 스크롤 버튼은 숨김. 하나의 전역 룰(`scrollbar-width: thin; scrollbar-color` + `::-webkit-scrollbar` 세트) — 컴포넌트별로 스타일링하지 않으며, 트랙 필이나 보더를 주지 않음. 네이티브 크롬 스크롤바는 보더 우선 규칙이 확장되지 *않는* 유일한 기본값입니다: 보더가 있는 트랙은 가끔만 스크롤되는 콘텐츠 주위에 영구적 프레임을 그립니다.
- 섀도는 페이지 위로 떠 있는 요소에만 예약됩니다: `raised`(스티키 바), `overlay`(드롭다운, 팝오버, 툴팁 — v4.0 이후 부드럽고 큰 블러), `modal`(다이얼로그, 드로어). 정적 카드, 버튼, 입력에는 섀도를 NEVER 넣지 마십시오(Card `elevated` 제외).
- 승인된 예외: `border.*` 토큰을 쓰는 `inset 0 0 0 1px` 링은 보더 대체입니다(진짜 보더가 레이아웃을 밀어낼 곳에 사용, 예: 캘린더 날짜 셀) — 이것은 엘리베이션이 아니며 블러가 없습니다.
- **쌓임(Stacking, v5.1):** 떠 있거나 고정된 요소는 정확히 하나의 `--sy-z-*` 토큰을 취합니다 — `sticky`(100, 테이블 헤더, 툴바, 솔리드 Banner 스트립) < `dropdown`(200, 메뉴/팝오버/팔레트) < `drawer`(300) < `modal`(400) < `toast`(500) < `tooltip`(600). 임의의 z-index 값은 금지됩니다. 두 요소가 다투면 해법은 9999가 아니라 스케일입니다.

래디우스: 입력/버튼/칩 `sm`(8px) · 카드/패널/메뉴 `md`(10px) · 모달/드로어 `lg`(16px) · **섹션 셸 `xl`(24px, v6.9)** · 필/아바타 `full`. `xl`은 직사각형 컨테이너의 상한이며 섹션 셸에 예약됩니다 — 페이지 영역이 자리 잡는 가장 바깥의 둥근 웰(SplitPanel 컨테이너, 콘텐츠 웰, hero/empty 웰): 영역당 하나의 셸 레벨, 카드나 오버레이에는 절대 안 되며, 다른 셸 안에 절대 중첩하지 않음. 셸은 `space-24` 이상을 패딩하여, 동심원 코너 규칙이 자식에 걸리지 않게 합니다.

**Glass 머티리얼(v6.22, 관할 v6.31 수정):** glass는 패널 뒤에 라이브 콘텐츠가 있는 곳에 속합니다 — 스크림 위의 블러는 이미 어두워진 백드롭을 샘플링해 구조상 탁한 회색으로 읽힙니다(v6.22의 스크림 게이팅은 물리를 거꾸로 잡았고, 관리자가 Template Library에서 이를 잡아냄). 관할: **스크림 없는 떠 있는 레이어에만 — CommandPalette(스크림을 버림; 프로스트 자체가 포커스 장치, esc/클릭어웨이는 여전히 닫음)와 Composer의 후속 패널.** 스크림이 걸린 차단 레이어(Modal, Drawer)는 불투명 `bg.raised`입니다 — 스크림이 이미 강조를 낮춥니다. 해부: `glass.surface` 필(불투명도 하한 0.82로 사전 검증된 텍스트 쌍을 근사적으로 유효하게 유지) + `backdrop-filter: blur(glass.blur)` + 라이트 모드 `border.default` 헤어라인(다크는 `border.overlay` 유지) + 레이어의 표준 섀도. 메뉴, 팝오버, 툴팁, 토스트는 불투명하게 유지됩니다 — 블러 비용은 스크롤/포인터 상호작용 중에 누적되며, 새로운 glass 표면은 거버넌스 제안을 요합니다. `prefers-reduced-transparency: reduce`는 불투명으로 폴백합니다. `backdrop-filter`는 이 형태로만 합법입니다(SY015).

**동심원 코너(concentric-corner) 규칙(v6.8, 필수):** 둥근 요소가 둥근 컨테이너의 코너 영역에 놓일 때 — 인셋이 바깥 래디우스보다 작아 두 곡선이 시각적으로 상호작용함 — 안쪽 래디우스는 바깥 래디우스에서 인셋을 뺀 값(`inner = outer − inset`)과 MUST 같아야 합니다. 불일치한 중첩이 컨트롤을 미묘하게 "어긋나" 보이게 만드는 원인입니다. 두 래디우스 모두 스케일에서 나와야 합니다. 뺄셈 결과가 스케일을 벗어나면 래디우스가 아니라 *인셋*을 조정하십시오(SegmentedControl: 8 − 4px 패딩 = 4; Menu: 10 − 6px 패딩 = 4). 인셋 0은 래디우스를 정확히 공유합니다(플러시 중첩: 타이트한 Card 헤더, ProposalCard 밴드). 바깥 래디우스 이상으로 인셋된 요소(예: 카드 안 16px에 위치한 diff 블록)는 면제됩니다 — 코너가 더 이상 상호작용하지 않습니다. 자유롭게 흐르는 콘텐츠(Combobox 트리거 안에서 줄바꿈되는 칩)는 면제됩니다. 이 규칙은 코너에 고정된 구조에 적용됩니다: 세그먼트, 메뉴 항목, 밴드 헤더, 부착된 그룹.

## 6. 모션

모션은 인과를 확인시켜 줍니다. 결코 장식하지 않습니다. 지속 시간: `instant` 100ms(hover, focus) · `fast` 150ms(드롭다운, 툴팁) · `base` 200ms(모달, 드로어, 아코디언) · `slow` 300ms(페이지 수준 전환, 토스트).

**마무리 규칙(Finish rule, v4.0, 필수):** 모든 상호작용 요소는 배경, border-color, color를 `instant`–`fast`에 `standard` 이징으로 전환합니다 — hover와 pressed 상태는 NEVER 뚝 끊기지 않습니다. 승인된 미세 처리: 상호작용 Card에만 `hover-lift` — translateY(−1px) + `shadow.raised`를 `fast`에; 어디에도 스케일링이나 바운스 없음.

**진입 규칙(Entrance rule, v6.6, 필수):** 떠 있는 레이어는 4px 이동 + 페이드로 진입합니다 — 메뉴/팝오버/팔레트는 상승(`translateY(4px)→0`)을 `base`에; 토스트는 오른쪽에서 슬라이드 인(`translateX(8px)→0`)을 `slow`에; 툴팁은 페이드만. 어떤 진입도 한 지속 시간 단계를 넘지 않으며, 어떤 것도 "튀지" 않습니다. 퇴장은 `fast`의 단순 페이드입니다 — 떠나기는 도착보다 빠르게 느껴져야 합니다.

- 이징: 대부분 `standard`; 나타나는 요소는 `enter`; 떠나는 요소는 `exit`.
- `opacity`와 `transform`만 애니메이션합니다. 레이아웃 속성(width/height/top/left)은 아코디언 높이를 제외하고 NEVER 애니메이션하지 마십시오.
- 큰 데이터 영역(테이블 정렬, 필터링 결과)은 NEVER 애니메이션하지 마십시오 — 데이터 업데이트는 뚝 바뀝니다.
- `prefers-reduced-motion`을 존중하십시오: 모든 비필수 모션은 ≤100ms 불투명도 페이드로 축소됩니다.

## 7. 아이코노그래피

- 단일 아이콘 패밀리, 스트로크 기반, 20px 그리드에서 1.5px 스트로크(Lucide). 사용 가능한 집합은 **`icons.md`의 닫힌 개념→아이콘 레지스트리**(v5.1)입니다 — 등록되지 않은 개념에는 아이콘이 없고, 등록되지 않은 Lucide 이름은 위반입니다.
- 크기: 16px(인라인, 밀집 컨트롤), 20px(기본 컨트롤, 내비게이션), 24px(empty state, 기능 모먼트). 다른 크기는 없음.
- 아이콘은 자신의 맥락의 텍스트 색상을 상속합니다. 아이콘 전용 버튼은 접근성 레이블을 MUST 가지며, 승인된 아이콘 액션 목록(`components.md` §Button)의 액션에만 허용됩니다.
- 이모지를 UI 아이코노그래피로 NEVER 사용하지 마십시오.

## 8. 접근성 베이스라인

**문서화된 편차 — 솔리드 상태 레이블.** 관리자의 명시적 결정(v3.5–3.7)에 따라, `success`/`warning`/`danger` 솔리드 필 위 흰색 텍스트는 약 3.4–3.5:1로 동작합니다 — 일반 텍스트에 대한 AA의 4.5:1 미만, 3:1 하드 하한 초과. 이 편차는 한정됩니다: 솔리드 Badge 레이블, 솔리드 Banner 스트립, danger Button 정지 상태에만 적용됩니다(짧은 레이블, 문장이나 본문 텍스트 아님; danger hover는 AA로 다시 어두워짐). 검증기는 이 쌍에 ≥3:1을 시행하는 한편 나머지 모든 곳에서 4.5:1을 유지합니다. **웨이트 보정은 필수입니다:** 4.5:1 미만으로 동작하는 솔리드 필 위 텍스트는 최소 세미볼드(600)입니다 — 낮은 대비는 가는 스트로크를 가장 심하게 벌합니다(배지: 기본 `micro`/600; `lg` 솔리드 배지는 600으로 상향; 솔리드 Banner 텍스트는 `body-sm` regular이 아니라 13px 세미볼드; danger Button 레이블은 정지 필이 약 3.4:1로 동작하므로 600). AA를 통과하는 솔리드 필(primary, accent)은 정상 레이블 웨이트를 유지합니다. 염두에 둘 결과: 이 항목은 모든 공식 WCAG/VPAT 감사에서 드러날 것입니다. 엄격한 준수가 요구되면 이 두 필을 600 수준의 프리미티브로 되돌리십시오.

그 외에는 모든 것에 대해 WCAG 2.1 AA가 최저 기준입니다: 대비(§1.2 참조), 보이는 포커스(2px `border.focus` 링, 2px 오프셋, 모든 상호작용 요소에 — 절대 제거하지 않고 이 규칙 안에서만 재스타일링), 완전한 키보드 조작성, 밀집 모드에서도 포인터 타겟 ≥ 24×24px(WCAG 2.5.8; 밀집 24px 컨트롤이 이를 정확히 충족 — 더 줄이지 말 것), 스크린 리더가 합성 언어를 전환할 수 있도록 영역별 올바른 `lang` 속성.

**포커스 관리(v6.1).** 오버레이 유형별로 결정론적:

- 열기: Modal → 가장 덜 파괴적인 조작 가능 컨트롤(확인 다이얼로그에서는 Cancel — 절대 danger 버튼 아님); Drawer → 그 헤딩; Menu/Popover/ContextMenu → 첫 항목; CommandPalette → 그 입력; Popconfirm → Cancel.
- 닫기: 포커스는 항상 트리거 요소로 돌아갑니다(사라졌다면 가장 가까운 생존 조상으로 — 예: 행 삭제 후에는 다음 행).
- Modal과 Drawer는 포커스를 트랩합니다. Esc는 z-스케일의 최상위 레이어만 닫습니다.
- 복합 위젯(Menu, Tree, Table 그리드 내비, ChoiceCard 그룹, SegmentedControl)에서의 로빙 tabindex: 위젯당 하나의 Tab 스톱, 화살표로 안에서 이동.
- 스켈레톤과 disabled 영역은 포커스 불가입니다. 포커스는 placeholder에 절대 안착하지 않습니다.

## 9. 키보드 단축키 (v6.1 — 닫힌 레지스트리)

전역 단축키는 닫힌 집합입니다. 추가는 제안 대상입니다:

| 단축키 | 액션 |
|---|---|
| ⌘K / Ctrl+K | CommandPalette |
| ⌘Enter | 전송(Composer, 여러 줄 입력으로 인해 그냥 Enter가 모호해질 때 — 제품 설정) |
| Esc | 최상위 레이어 닫기 / 편집 취소 / 선택 해제 (이 우선순위로) |
| ⌘/ | 단축키 참조 오버레이 |

규칙: 단일 문자 전역 단축키 없음(타이핑 및 IME 조합과 충돌함); 브라우저 예약 조합(⌘L, ⌘T, ⌘W, ⌘R…)은 NEVER 오버라이드하지 않음; 컴포넌트 범위 키(화살표, Enter, Space, 타입어헤드)는 각 컴포넌트 스펙에 정의되며 전역으로 등록되지 않음; 모든 단축키는 kbd 슬롯을 통해 Tooltip에 그리고 ⌘/ 오버레이에 드러남; 단축키는 두 로케일에서 동일함(kbd 심볼은 로케일화되지 않음).
